There are two different kinds of package migrations: [待校准@2000]

  • Migrating the source code of an existing package from ROS 1 to ROS 2 with the intent that a significant part of the source code will stay the same or at least similar. An example for this could be pluginlib where the source code is maintained in different branches within the same repository and commonly patches can be ported between those branches when necessary. [待校准@2001]

  • Implementing the same or similar functionality of a ROS 1 package for ROS 2 but with the assumption that the source code will be significantly different. An example for this could be roscpp in ROS 1 and rclcpp in ROS 2 which are separate repositories and don't share any code. [待校准@2002]

This article focuses on the former case and describes the high-level steps to migrate a ROS 1 package to ROS 2. It does not aim to be a step-by-step migration instruction and is not considered the final "solution". Future versions will aim to make migration smoother and less effort up to the point of maintaining a single package from the same branch for ROS 1 as well as ROS 2. [待校准@2003]

Prerequisites [待校准@2004]

Before being able to migrate a ROS 1 package to ROS 2 all of its dependencies must be available in ROS 2. [待校准@2005]

Migration steps [待校准@2006]

Package manifests [待校准@2007]

ROS 2 doesn't support format 1 of the package specification but only newer format versions (2 and higher). Therefore the package.xml file must be updated to at least format 2 if it uses format 1. Since ROS 1 supports all formats it is safe to perform that conversion in the ROS 1 package. [待校准@2008]

Some packages might have different names in ROS 2 so the dependencies might need to be updated accordingly. [待校准@2009]

Metapackages [待校准@2010]

ROS 2 doesn't have a special package type for metapackages. Metapackages can still exist as regular packages that only contain runtime dependencies. When migrating metapackages from ROS 1, simply remove the <metapackage /> tag in your package manifest. [待校准@2011]

Message, service, and action definitions [待校准@2012]

Message files must end in .msg and must be located in the subfolder msg. Service files must end in .srv and must be located in the subfolder srv. Actions files must end in .action and must be located in the subfolder action. [待校准@2013]

These files might need to be updated to comply with the ROS Interface definition. Some primitive types have been removed and the types duration and time which were builtin types in ROS 1 have been replaced with normal message definitions and must be used from the builtin_interfaces package. Also some naming conventions are stricter than in ROS 1. [待校准@2014]

In your package.xml: [待校准@2015]

  • Add <buildtool_depend>rosidl_default_generators</buildtool_depend>. [待校准@2016]

  • Add <exec_depend>rosidl_default_runtime</exec_depend>. [待校准@2017]

  • For each dependent message package, add <depend>message_package</depend>. [待校准@2018]

In your CMakeLists.txt: [待校准@2019]

  • Start by enabling C++14 [待校准@2020]

set(CMAKE_CXX_STANDARD 14)

  • Add find_package(rosidl_default_generators REQUIRED) [待校准@2021]

  • For each dependent message package, add find_package(message_package REQUIRED) and replace the CMake function call to generate_messages with rosidl_generate_interfaces. [待校准@2022]

This will replace add_message_files and add_service_files listing of all the message and service files, which can be removed. [待校准@2023]

Build system [待校准@2024]

The build system in ROS 2 is called ament and the build tool is colcon. Ament is built on CMake: ament_cmake provides CMake functions to make writing CMakeLists.txt files easier. [待校准@2025]

Build tool [待校准@2026]

Instead of using catkin_make, catkin_make_isolated or catkin build ROS 2 uses the command line tool colcon to build and install a set of packages. [待校准@2027]

Pure Python package [待校准@2028]

If the ROS 1 package uses CMake only to invoke the setup.py file and does not contain anything beside Python code (e.g. also no messages, services, etc.) it should be converted into a pure Python package in ROS 2: [待校准@2029]

  • Update or add the build type in the package.xml file: [待校准@2030]

    <export>
  <build_type>ament_python</build_type>
</export>

  • Remove the CMakeLists.txt file [待校准@2031]

  • Update the setup.py file to be a standard Python setup script [待校准@2032]

ROS 2 supports Python 3 only. While each package can choose to also support Python 2 it must invoke executables with Python 3 if it uses any API provided by other ROS 2 packages. [待校准@2033]

Update the CMakeLists.txt to use ament_cmake [待校准@2034]

Apply the following changes to use ament_cmake instead of catkin: [待校准@2035]

  • Set the build type in the package.xml file export section: [待校准@2036]

    <export>
  <build_type>ament_cmake</build_type>
</export>

  • Replace the find_package invocation with catkin and the COMPONENTS with: [待校准@2037]

    find_package(ament_cmake REQUIRED)
find_package(component1 REQUIRED)
# ...
find_package(componentN REQUIRED)

  • Move and update the catkin_package invocation with: [待校准@2038]

    • Invoke ament_package instead but after all targets have been registered. [待校准@2039]

    • The only valid argument for ament_package is CONFIG_EXTRAS. All other arguments are covered by separate functions which all need to be invoked before ament_package: [待校准@2040]

      • Instead of passing CATKIN_DEPENDS ... call ament_export_dependencies(...) before. [待校准@2041]

      • Instead of passing INCLUDE_DIRS ... call ament_export_include_directories(...) before. [待校准@2042]

      • Instead of passing LIBRARIES ... call ament_export_libraries(...) before. [待校准@2043]

    • TODO document ament_export_targets (``ament_export_interfaces`` in Eloquent and older)? [待校准@2044]

  • Replace the invocation of add_message_files, add_service_files and generate_messages with rosidl_generate_interfaces. [待校准@2045]

    • The first argument is the target_name. If you're building just one library it's ${PROJECT_NAME} [待校准@2046]

    • Followed by the list of message filenames, relative to the package root. [待校准@2047]

      • If you will be using the list of filenames multiple times, it is recommended to compose a list of message files and pass the list to the function for clarity. [待校准@2048]

    • The final multi-value-keyword argument fpr generate_messages is DEPENDENCIES which requires the list of dependent message packages. [待校准@2049]

      rosidl_generate_interfaces(${PROJECT_NAME}
  ${msg_files}
  DEPENDENCIES std_msgs
)

  • Remove any occurrences of the devel space. Related CMake variables like CATKIN_DEVEL_PREFIX do not exist anymore. [待校准@2050]

    • The CATKIN_DEPENDS and DEPENDS arguments are passed to the new function ament_export_dependencies. [待校准@2051]

    • CATKIN_GLOBAL_BIN_DESTINATION: bin [待校准@2052]

    • CATKIN_GLOBAL_INCLUDE_DESTINATION: include [待校准@2053]

    • CATKIN_GLOBAL_LIB_DESTINATION: lib [待校准@2054]

    • CATKIN_GLOBAL_LIBEXEC_DESTINATION: lib [待校准@2055]

    • CATKIN_GLOBAL_SHARE_DESTINATION: share [待校准@2056]

    • CATKIN_PACKAGE_BIN_DESTINATION: lib/${PROJECT_NAME} [待校准@2057]

    • CATKIN_PACKAGE_INCLUDE_DESTINATION: include/${PROJECT_NAME} [待校准@2058]

    • CATKIN_PACKAGE_LIB_DESTINATION: lib [待校准@2059]

    • CATKIN_PACKAGE_SHARE_DESTINATION: share/${PROJECT_NAME} [待校准@2060]

Unit tests [待校准@2061]

If you are using gtest: [待校准@2062]

Replace CATKIN_ENABLE_TESTING with BUILD_TESTING. Replace catkin_add_gtest with ament_add_gtest. [待校准@2063]

-   if (CATKIN_ENABLE_TESTING)
-     find_package(GTest REQUIRED)  # or rostest
-     include_directories(${GTEST_INCLUDE_DIRS})
-     catkin_add_gtest(${PROJECT_NAME}-some-test src/test/some_test.cpp)
-     target_link_libraries(${PROJECT_NAME}-some-test
-       ${PROJECT_NAME}_some_dependency
-       ${catkin_LIBRARIES}
-       ${GTEST_LIBRARIES})
-   endif()
+   if (BUILD_TESTING)
+     find_package(ament_cmake_gtest REQUIRED)
+     ament_add_gtest(${PROJECT_NAME}-some-test src/test/test_something.cpp)
+     ament_target_dependencies(${PROJECT_NAME)-some-test
+       "rclcpp"
+       "std_msgs")
+     target_link_libraries(${PROJECT_NAME}-some-test

Messages, services, and actions [待校准@2072]

The namespace of ROS 2 messages, services, and actions use a subnamespace (msg, srv, or action, respectively) after the package name. Therefore an include looks like: #include <my_interfaces/msg/my_message.hpp>. The C++ type is then named: my_interfaces::msg::MyMessage. [待校准@2073]

Shared pointer types are provided as typedefs within the message structs: my_interfaces::msg::MyMessage::SharedPtr as well as my_interfaces::msg::MyMessage::ConstSharedPtr. [待校准@2074]

For more details please see the article about the generated C++ interfaces. [待校准@2075]

The migration requires includes to change by: [待校准@2076]

  • inserting the subfolder msg between the package name and message datatype [待校准@2077]

  • changing the included filename from CamelCase to underscore separation [待校准@2078]

  • changing from *.h to *.hpp [待校准@2079]

// ROS 1 style is in comments, ROS 2 follows, uncommented.
// # include <geometry_msgs/PointStamped.h>
#include <geometry_msgs/msg/point_stamped.hpp>

// geometry_msgs::PointStamped point_stamped;
geometry_msgs::msg::PointStamped point_stamped;

The migration requires code to insert the msg namespace into all instances. [待校准@2080]

Use of service objects [待校准@2081]

Service callbacks in ROS 2 do not have boolean return values. Instead of returning false on failures, throwing exceptions is recommended. [待校准@2082]

// ROS 1 style is in comments, ROS 2 follows, uncommented.
// #include "nav_msgs/GetMap.h"
#include "nav_msgs/srv/get_map.hpp"

// bool service_callback(
//   nav_msgs::GetMap::Request & request,
//   nav_msgs::GetMap::Response & response)
void service_callback(
  const std::shared_ptr<nav_msgs::srv::GetMap::Request> request,
  std::shared_ptr<nav_msgs::srv::GetMap::Response> response)
{
  // ...
  // return true;  // or false for failure
}

Example: Converting an existing ROS 1 package to use ROS 2 [待校准@2120]

Let's say that we have simple ROS 1 package called talker that uses roscpp in one node, called talker. This package is in a catkin workspace, located at ~/ros1_talker. [待校准@2121]

The ROS 1 code [待校准@2122]

Here's the directory layout of our catkin workspace: [待校准@2123]

$ cd ~/ros1_talker
$ find .
.
./src
./src/talker
./src/talker/package.xml
./src/talker/CMakeLists.txt
./src/talker/talker.cpp

Here is the content of those three files: [待校准@2124]

src/talker/package.xml: [待校准@2125]

<package>
  <name>talker</name>
  <version>0.0.0</version>
  <description>talker</description>
  <maintainer email="gerkey@osrfoundation.org">Brian Gerkey</maintainer>
  <license>Apache 2.0</license>
  <buildtool_depend>catkin</buildtool_depend>
  <build_depend>roscpp</build_depend>
  <build_depend>std_msgs</build_depend>
  <run_depend>roscpp</run_depend>
  <run_depend>std_msgs</run_depend>

src/talker/CMakeLists.txt: [待校准@2126]

cmake_minimum_required(VERSION 2.8.3)
project(talker)
find_package(catkin REQUIRED COMPONENTS roscpp std_msgs)
catkin_package()
include_directories(${catkin_INCLUDE_DIRS})
add_executable(talker talker.cpp)

install(TARGETS talker
  RUNTIME DESTINATION ${CATKIN_PACKAGE_BIN_DESTINATION})

src/talker/talker.cpp: [待校准@2127]

#include <sstream>
#include "ros/ros.h"
#include "std_msgs/String.h"
int main(int argc, char **argv)
{
  ros::init(argc, argv, "talker");
  ros::NodeHandle n;
  ros::Publisher chatter_pub = n.advertise<std_msgs::String>("chatter", 1000);
  ros::Rate loop_rate(10);
  int count = 0;
  std_msgs::String msg;
  while (ros::ok())
  {
    std::stringstream ss;
    ss << "hello world " << count++;
    msg.data = ss.str();
    ROS_INFO("%s", msg.data.c_str());
    chatter_pub.publish(msg);
    ros::spinOnce();
    loop_rate.sleep();
  }
  return 0;
}

Migrating to ROS 2 [待校准@2133]

Let's start by creating a new workspace in which to work: [待校准@2134]

mkdir ~/ros2_talker
cd ~/ros2_talker

We'll copy the source tree from our ROS 1 package into that workspace, where we can modify it: [待校准@2135]

mkdir src
cp -a ~/ros1_talker/src/talker src

Now we'll modify the C++ code in the node. The ROS 2 C++ library, called rclcpp, provides a different API from that provided by roscpp. The concepts are very similar between the two libraries, which makes the changes reasonably straightforward to make. [待校准@2136]

Included headers [待校准@2137]

In place of ros/ros.h, which gave us access to the roscpp library API, we need to include rclcpp/rclcpp.hpp, which gives us access to the rclcpp library API: [待校准@2138]

//#include "ros/ros.h"
#include "rclcpp/rclcpp.hpp"

To get the std_msgs/String message definition, in place of std_msgs/String.h, we need to include std_msgs/msg/string.hpp: [待校准@2139]

//#include "std_msgs/String.h"
#include "std_msgs/msg/string.hpp"

Changing C++ library calls [待校准@2140]

Instead of passing the node's name to the library initialization call, we do the initialization, then pass the node name to the creation of the node object (we can use the auto keyword because now we're requiring a C++14 compiler): [待校准@2141]

//  ros::init(argc, argv, "talker");
//  ros::NodeHandle n;
    rclcpp::init(argc, argv);
    auto node = rclcpp::Node::make_shared("talker");

The creation of the publisher and rate objects looks pretty similar, with some changes to the names of namespace and methods. [待校准@2142]

//  ros::Publisher chatter_pub = n.advertise<std_msgs::String>("chatter", 1000);
//  ros::Rate loop_rate(10);
  auto chatter_pub = node->create_publisher<std_msgs::msg::String>("chatter",
    1000);
  rclcpp::Rate loop_rate(10);

To further control how message delivery is handled, a quality of service (QoS) profile could be passed in. The default profile is rmw_qos_profile_default. For more details, see the design document and concept overview. [待校准@2143]

The creation of the outgoing message is different in the namespace: [待校准@2144]

//  std_msgs::String msg;
  std_msgs::msg::String msg;

In place of ros::ok(), we call rclcpp::ok(): [待校准@2145]

//  while (ros::ok())
  while (rclcpp::ok())

Inside the publishing loop, we access the data field as before: [待校准@2146]

msg.data = ss.str();

To print a console message, instead of using ROS_INFO(), we use RCLCPP_INFO() and its various cousins. The key difference is that RCLCPP_INFO() takes a Logger object as the first argument. [待校准@2147]

//    ROS_INFO("%s", msg.data.c_str());
    RCLCPP_INFO(node->get_logger(), "%s\n", msg.data.c_str());

Publishing the message is the same as before: [待校准@2148]

chatter_pub->publish(msg);

Spinning (i.e., letting the communications system process any pending incoming/outgoing messages) is different in that the call now takes the node as an argument: [待校准@2149]

//    ros::spinOnce();
    rclcpp::spin_some(node);

Sleeping using the rate object is unchanged. [待校准@2150]

Putting it all together, the new talker.cpp looks like this: [待校准@2151]

#include <sstream>
// #include "ros/ros.h"
#include "rclcpp/rclcpp.hpp"
// #include "std_msgs/String.h"
#include "std_msgs/msg/string.hpp"
int main(int argc, char **argv)
{
//  ros::init(argc, argv, "talker");
//  ros::NodeHandle n;
  rclcpp::init(argc, argv);
  auto node = rclcpp::Node::make_shared("talker");
//  ros::Publisher chatter_pub = n.advertise<std_msgs::String>("chatter", 1000);
//  ros::Rate loop_rate(10);
  auto chatter_pub = node->create_publisher<std_msgs::msg::String>("chatter", 1000);
  rclcpp::Rate loop_rate(10);
  int count = 0;
//  std_msgs::String msg;
  std_msgs::msg::String msg;
//  while (ros::ok())
  while (rclcpp::ok())
  {
    std::stringstream ss;
    ss << "hello world " << count++;
    msg.data = ss.str();
//    ROS_INFO("%s", msg.data.c_str());
    RCLCPP_INFO(node->get_logger(), "%s\n", msg.data.c_str());
    chatter_pub->publish(msg);
//    ros::spinOnce();
    rclcpp::spin_some(node);
    loop_rate.sleep();
  }
  return 0;
}

Changing the package.xml [待校准@2152]

ROS 2 doesn't support format 1 of the package specification but only newer format versions (2 and higher). We start by specifying the format version in the package tag: [待校准@2153]

<!-- <package> -->
<package format="2">

ROS 2 uses a newer version of catkin, called ament_cmake, which we specify in the buildtool_depend tag: [待校准@2154]

<!--  <buildtool_depend>catkin</buildtool_depend> -->
  <buildtool_depend>ament_cmake</buildtool_depend>

In our build dependencies, instead of roscpp we use rclcpp, which provides the C++ API that we use. [待校准@2155]

<!--  <build_depend>roscpp</build_depend> -->
  <build_depend>rclcpp</build_depend>

We make the same addition in the run dependencies and also update from the run_depend tag to the exec_depend tag (part of the upgrade to version 2 of the package format): [待校准@2156]

<!--  <run_depend>roscpp</run_depend> -->
  <exec_depend>rclcpp</exec_depend>
<!--  <run_depend>std_msgs</run_depend> -->
  <exec_depend>std_msgs</exec_depend>

In ROS 1, we use <depend> to simplify specifying dependencies for both compile-time and runtime. We can do the same in ROS 2: [待校准@2157]

<depend>rclcpp</depend>
<depend>std_msgs</depend>

We also need to tell the build tool what kind of package we are, so that it knows how to build us. Because we're using ament and CMake, we add the following lines to declare our build type to be ament_cmake: [待校准@2158]

<export>
  <build_type>ament_cmake</build_type>
</export>

Putting it all together, our package.xml now looks like this: [待校准@2159]

<!-- <package> -->
<package format="2">
  <name>talker</name>
  <version>0.0.0</version>
  <description>talker</description>
  <maintainer email="gerkey@osrfoundation.org">Brian Gerkey</maintainer>
  <license>Apache License 2.0</license>
<!--  <buildtool_depend>catkin</buildtool_depend> -->
  <buildtool_depend>ament_cmake</buildtool_depend>
<!--  <build_depend>roscpp</build_depend> -->
<!--  <run_depend>roscpp</run_depend> -->
<!--  <run_depend>std_msgs</run_depend> -->
  <depend>rclcpp</depend>
  <depend>std_msgs</depend>
  <export>
    <build_type>ament_cmake</build_type>
  </export>
</package>

Changing the CMake code [待校准@2161]

ROS 2 relies on a higher version of CMake: [待校准@2162]

#cmake_minimum_required(VERSION 2.8.3)
cmake_minimum_required(VERSION 3.5)

ROS 2 relies on the C++14 standard. Depending on what compiler you're using, support for C++14 might not be enabled by default. Using gcc 5.3 (which is what is used on Ubuntu Xenial), we need to enable it explicitly, which we do by adding this line near the top of the file: [待校准@2163]

set(CMAKE_CXX_STANDARD 14)

The preferred way to work on all platforms is this: [待校准@2164]

if(NOT CMAKE_CXX_STANDARD)
  set(CMAKE_CXX_STANDARD 14)
endif()
if(CMAKE_COMPILER_IS_GNUCXX OR CMAKE_CXX_COMPILER_ID MATCHES "Clang")
  add_compile_options(-Wall -Wextra -Wpedantic)
endif()

Using catkin, we specify the packages we want to build against by passing them as COMPONENTS arguments when initially finding catkin itself. With ament_cmake, we find each package individually, starting with ament_cmake: [待校准@2165]

#find_package(catkin REQUIRED COMPONENTS roscpp std_msgs)
find_package(ament_cmake REQUIRED)
find_package(rclcpp REQUIRED)
find_package(std_msgs REQUIRED)

System dependencies can be found as before: [待校准@2166]

find_package(Boost REQUIRED COMPONENTS system filesystem thread)

We call catkin_package() to auto-generate things like CMake configuration files for other packages that use our package. Whereas that call happens before specifying targets to build, we now call the analogous ament_package() after the targets: [待校准@2167]

# catkin_package()
# At the bottom of the file:
ament_package()

The only directories that need to be manually included are local directories and dependencies that are not ament packages: [待校准@2168]

#include_directories(${catkin_INCLUDE_DIRS})
include_directories(include ${Boost_INCLUDE_DIRS})

A better alternative is to specify include directories for each target individually, rather than including all the directories for all targets: [待校准@2169]

target_include_directories(target PUBLIC include ${Boost_INCLUDE_DIRS})

Similar to how we found each dependent package separately, we need to link each one to the build target. To link with dependent packages that are ament packages, instead of using target_link_libraries(), ament_target_dependencies() is a more concise and more thorough way of handling build flags. It automatically handles both the include directories defined in _INCLUDE_DIRS and linking libraries defined in _LIBRARIES. [待校准@2170]

#target_link_libraries(talker ${catkin_LIBRARIES})
ament_target_dependencies(talker
  rclcpp
  std_msgs)

To link with packages that are not ament packages, such as system dependencies like Boost, or a library being built in the same CMakeLists.txt, use target_link_libraries(): [待校准@2171]

target_link_libraries(target ${Boost_LIBRARIES})

For installation, catkin defines variables like CATKIN_PACKAGE_BIN_DESTINATION. With ament_cmake, we just give a path relative to the installation root, like bin for executables: [待校准@2172]

#install(TARGETS talker
#  RUNTIME DESTINATION ${CATKIN_PACKAGE_BIN_DESTINATION})
install(TARGETS talker
  DESTINATION lib/${PROJECT_NAME})

Optionally, we can install and export the included directories for downstream packages: [待校准@2173]

install(DIRECTORY include/
  DESTINATION include)
ament_export_include_directories(include)

Optionally, we can export dependencies for downstream packages: [待校准@2174]

ament_export_dependencies(std_msgs)

Putting it all together, the new CMakeLists.txt looks like this: [待校准@2175]

#cmake_minimum_required(VERSION 2.8.3)

endif()

find_package(rclcpp REQUIRED)

#catkin_package()

  rclcpp

#install(TARGETS talker

Update scripts [待校准@2181]

ROS CLI arguments [待校准@2182]

Since ROS Eloquent, ROS arguments should be scoped with --ros-args and a trailing -- (the trailing double dash may be elided if no arguments follow it). [待校准@2183]

Remapping names is similar to ROS 1, taking on the form from:=to, except that it must be preceded by a --remap (or -r) flag. For example: [待校准@2184]

ros2 run some_package some_ros_executable --ros-args -r foo:=bar

We use a similar syntax for parameters, using the --param (or -p) flag: [待校准@2185]

ros2 run some_package some_ros_executable --ros-args -p my_param:=value

Note, this is different than using a leading underscore in ROS 1. [待校准@2186]

To change a node name use __node (the ROS 1 equivalent is __name): [待校准@2187]

ros2 run some_package some_ros_executable --ros-args -r __node:=new_node_name

Note the use of the -r flag. The same remap flag is needed for changing the namespace __ns: [待校准@2188]

ros2 run some_package some_ros_executable --ros-args -r __ns:=/new/namespace

There is no equivalent in ROS 2 for the following ROS 1 keys: [待校准@2189]

  • __log (but --log-config-file can be used to provide a logger configuration file) [待校准@2190]

  • __ip [待校准@2191]

  • __hostname [待校准@2192]

  • __master [待校准@2193]

For more information, see the design document. [待校准@2194]

Licensing [待校准@2214]

In ROS 2 our recommended license is the Apache 2.0 License. In ROS 1 our recommended license was the 3-Clause BSD License. [待校准@2215]

For any new project we recommend using the Apache 2.0 License, whether ROS 1 or ROS 2. [待校准@2216]

However, when migrating code from ROS 1 to ROS 2 we cannot simply change the license. The existing license must be preserved for any preexisting contributions. [待校准@2217]

To that end if a package is being migrated we recommend keeping the existing license and continuing to contribute to that package under the existing OSI license, which we expect to be the BSD license for core elements. [待校准@2218]

This will keep things clear and easy to understand. [待校准@2219]

Changing the License [待校准@2220]

It is possible to change the license, however you will need to contact all the contributors and get permission. For most packages this is likely to be a significant effort and not worth considering. If the package has a small set of contributors then this may be feasible. [待校准@2221]
