Quality guide: ensuring code quality [待校准@2231]

Table of Contents [@1329]

  • Static code analysis as part of the ament package build [待校准@2234]

  • Static Thread Safety Analysis via Code Annotation [待校准@2257]

  • Dynamic analysis (data races & deadlocks) [待校准@2305]

This page gives guidance about how to improve the software quality of ROS 2 packages, focusing on more specific areas than the Quality Practices section of the Developer Guide. [待校准@2232]

The sections below intend to address ROS 2 core, application and ecosystem packages and the core client libraries, C++ and Python. The solutions presented are motivated by design and implementation considerations to improve quality attributes like "Reliability", "Security", "Maintainability", "Determinism", etc. which relate to non-functional requirements. [待校准@2233]

Static code analysis as part of the ament package build [待校准@2234]

Context: [待校准@2235]

  • You have developed your C++ production code. [待校准@2236]

  • You have created a ROS 2 package with build support with ament. [待校准@2237]

Problem: [待校准@2238]

  • Library level static code analysis is not run as part of the package build procedure. [待校准@2239]

  • Library level static code analysis needs to be executed manually. [待校准@2240]

  • Risk of forgetting to execute library level static code analysis before building a new package version. [待校准@2241]

Solution: [待校准@2242]

  • Use the integration capabilities of ament to execute static code analysis as part of the package build procedure. [待校准@2243]

Implementation: [待校准@2244]

  • Insert into the packages CMakeLists.txt file. [待校准@2245]

...
if(BUILD_TESTING)
  find_package(ament_lint_auto REQUIRED)
  ament_lint_auto_find_test_dependencies()
  ...
endif()
...

  • Insert the ament_lint test dependencies into the packages package.xml file. [待校准@2246]

...
<package format="2">
  ...
  <test_depend>ament_lint_auto</test_depend>
  <test_depend>ament_lint_common</test_depend>
  ...
</package>

Examples: [待校准@2247]

  • rclcpp: [待校准@2248]

    • rclcpp/rclcpp/CMakeLists.txt [待校准@2249]

    • rclcpp/rclcpp/package.xml [待校准@2250]

  • rclcpp_lifecycle: [待校准@2251]

    • rclcpp/rclcpp_lifecycle/CMakeLists.txt [待校准@2252]

    • rclcpp/rclcpp_lifecycle/package.xml [待校准@2253]

Resulting context: [待校准@2254]

  • The static code analysis tools supported by ament are run as part of the package build. [待校准@2255]

  • Static code analysis tools not supported by ament need to be executed separately. [待校准@2256]

Static Thread Safety Analysis via Code Annotation [待校准@2257]

Context: [待校准@2258]

  • You are developing/debugging your multithreaded C++ production code [待校准@2259]

  • You access data from multiple threads in C++ code [待校准@2260]

Problem: [待校准@2261]

  • Data races and deadlocks can lead to critical bugs. [待校准@2262]

Solution: [待校准@2263]

  • Utilize Clang's static Thread Safety Analysis by annotating threaded code [待校准@2264]

Context For Implementation: [待校准@2265]

To enable Thread Safety Analysis, code must be annotated to let the compiler know more about the semantics of the code. These annotations are Clang-specific attributes - e.g. __attribute__(capability())). Instead of using those attributes directly, ROS 2 provides preprocessor macros that are erased when using other compilers. [待校准@2266]

These macros can be found in rcpputils/thread_safety_annotations.hpp [待校准@2267]

The Thread Safety Analysis documentation states [待校准@2268]

Thread safety analysis can be used with any threading library, but it does require that the threading API be wrapped in classes and methods which have the appropriate annotations [待校准@2269]

We have decided that we want ROS 2 developers to be able to use std:: threading primitives directly for their development. We do not want to provide our own wrapped types as is suggested above. [待校准@2270]

There are three C++ standard libraries to be aware of [待校准@2271]

  • The GNU standard library libstdc++ - default on Linux, explicitly via the compiler option -stdlib=libstdc++ [待校准@2272]

  • The LLVM standard library libc++ (also called libcxx ) - default on macOS, explicitly set by the compiler option -stdlib=libc++ [待校准@2273]

  • The Windows C++ Standard Library - not relevant to this use case [待校准@2274]

libcxx annotates its std::mutex and std::lock_guard implementations for Thread Safety Analysis. When using GNU libstdc++ , those annotations are not present, so Thread Safety Analysis cannot be used on non-wrapped std:: types. [待校准@2275]

Therefore, to use Thread Safety Analysis directly with std:: types, we must use libcxx [待校准@2276]

Implementation: [待校准@2277]

The code migration suggestions here are by no means complete - when writing (or annotating existing) threaded code, you are encouraged to utilize as many of the annotations as is logical for your use case. However, this step-by-step is a great place to start! [待校准@2278]

  • Enabling Analysis for Package/Target [待校准@2279]

    When the C++ compiler is Clang, enable the -Wthread-safety flag. Example below for CMake-based projects [待校准@2280]

    if(CMAKE_CXX_COMPILER_ID MATCHES "Clang")
  add_compile_options(-Wthread-safety)   # for your whole package
  target_compile_options(${MY_TARGET} PUBLIC -Wthread-safety)  # for a single library or executable
endif()

  • Annotating Code [待校准@2281]

    • Step 1 - Annotate data members [待校准@2282]

      • Find anywhere that std::mutex is used to protect some member data [待校准@2283]

      • Add the RCPPUTILS_TSA_GUARDED_BY(mutex_name) annotation to the data that is protected by the mutex [待校准@2284]

      class Foo {
public:
  void incr(int amount) {
    std::lock_guard<std::mutex> lock(mutex_);
    bar += amount;
  }

  void get() const {
    return bar;
  }

private:
  mutable std::mutex mutex_;
  int bar RCPPUTILS_TSA_GUARDED_BY(mutex_) = 0;
};

    • Step 2 - Fix Warnings [待校准@2285]

      • In the above example - Foo::get will produce a compiler warning! To fix it, lock before returning bar [待校准@2286]

      void get() const {
  std::lock_guard<std::mutex> lock(mutex_);
  return bar;
}

    • Step 3 - (Optional but Recommended) Refactor Existing Code to Private-Mutex Pattern [待校准@2287]

      A recommended pattern in threaded C++ code is to always keep your mutex as a private: member of the data structure. This makes data safety the concern of the containing structure, offloading that responsibility from users of the structure and minimizing the surface area of affected code. [待校准@2288]

      Making your locks private may require rethinking the interfaces to your data. This is a great exercise - here are a few things to consider [待校准@2289]

      • You may want to provide specialized interfaces for performing analysis that requires complex locking logic, e.g. counting members in a filtered set of a mutex-guarded map structure, instead of actually returning the underlying structure to consumers [待校准@2290]

      • Consider copying to avoid blocking, where the amount of data is small. This can let other threads get on with accessing the shared data, which can potentially lead to better overall performance. [待校准@2291]

    • Step 4 - (Optional) Enable Negative Capability Analysis [待校准@2292]

      https://clang.llvm.org/docs/ThreadSafetyAnalysis.html#negative-capabilities [待校准@2293]

      Negative Capability Analysis lets you specify “this lock must not be held when calling this function”. It can reveal potential deadlock cases that other annotations cannot. [待校准@2294]

      • Where you specified -Wthread-safety, add the additional flag -Wthread-safety-negative [待校准@2295]

      • On any function that acquires a lock, use the RCPPUTILS_TSA_REQUIRES(!mutex) pattern [待校准@2296]

  • How to run the analysis [待校准@2297]

    • The ROS CI build farm runs a nightly job with libcxx, which will surface any issues in the ROS 2 core stack by being marked "Unstable" when Thread Safety Analysis raises warnings [待校准@2298]

    • For local runs, you have the following options, all equivalent [待校准@2299]

      • Use the colcon clang-libcxx mixin (see the documentation for configuring mixins) :: [待校准@2300]

        colcon build --mixin clang-libcxx

      • Passing compiler to CMake :: [待校准@2301]

        colcon build --cmake-args -DCMAKE_C_COMPILER=clang -DCMAKE_CXX_COMPILER=clang++ -DCMAKE_CXX_FLAGS='-stdlib=libc++ -D_LIBCPP_ENABLE_THREAD_SAFETY_ANNOTATIONS' -DFORCE_BUILD_VENDOR_PKG=ON --no-warn-unused-cli

      • Overriding system compiler :: [待校准@2302]

        CC=clang CXX=clang++ colcon build --cmake-args -DCMAKE_CXX_FLAGS='-stdlib=libc++ -D_LIBCPP_ENABLE_THREAD_SAFETY_ANNOTATIONS' -DFORCE_BUILD_VENDOR_PKG=ON --no-warn-unused-cli

Resulting Context: [待校准@2303]

  • Potential deadlocks and race conditions will be surfaced at compile time, when using Clang and libcxx [待校准@2304]

Dynamic analysis (data races & deadlocks) [待校准@2305]

Context: [待校准@2258]

  • You are developing/debugging your multithreaded C++ production code. [待校准@2306]

  • You use pthreads or C++11 threading + llvm libc++ (in case of ThreadSanitizer). [待校准@2307]

  • You do not use Libc/libstdc++ static linking (in case of ThreadSanitizer). [待校准@2308]

  • You do not build non-position-independent executables (in case of ThreadSanitizer). [待校准@2309]

Problem: [待校准@2261]

  • Data races and deadlocks can lead to critical bugs. [待校准@2262]

  • Data races and deadlocks cannot be detected using static analysis (reason: limitation of static analysis). [待校准@2310]

  • Data races and deadlocks must not show up during development debugging / testing (reason: usually not all possible control paths through production code exercised). [待校准@2311]

Solution: [待校准@2263]

  • Use a dynamic analysis tool which focuses on finding data races and deadlocks (here clang ThreadSanitizer). [待校准@2312]

Implementation: [待校准@2277]

  • Compile and link the production code with clang using the option -fsanitize=thread (this instruments the production code). [待校准@2313]

  • In case different production code shall be executed during analysis consider conditional compilation e.g. ThreadSanitizers _has_feature(thread_sanitizer). [待校准@2314]

  • In case some code shall not be instrumented consider ThreadSanitizers _/*attribute*/_((no_sanitize("thread"))). [待校准@2315]

  • In case some files shall not be instrumented consider file or function-level exclusion ThreadSanitizers blacklisting, more specific: ThreadSanitizers Sanitizer Special Case List or with ThreadSanitizers no_sanitize("thread") and use the option --fsanitize-blacklist. [待校准@2316]

Resulting context: [待校准@2317]

  • Higher chance to find data races and deadlocks in production code before deploying it. [待校准@2318]

  • Analysis result may lack reliability, tool in beta phase stage (in case of ThreadSanitizer). [待校准@2319]

  • Overhead due to production code instrumentation (maintenance of separate branches for instrumented/not instrumented production code, etc.). [待校准@2320]

  • Instrumented code needs more memory per thread (in case of ThreadSanitizer). [待校准@2321]

  • Instrumented code maps a lot virtual address space (in case of ThreadSanitizer). [待校准@2322]
