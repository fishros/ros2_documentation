    • Style [待校准@8069]

In order to achieve a consistent looking product we will all follow externally (if possible) defined style guidelines for each language. For other things like package layout or documentation layout we will need to come up with our own guidelines, drawing on current, popular styles in use now. [待校准@8063]

Additionally, wherever possible, developers should use integrated tools to allow them to check that these guidelines are followed in their editors. For example, everyone should have a PEP8 checker built into their editor to cut down on review iterations related to style. [待校准@8064]

Also where possible, packages should check style as part of their unit tests to help with the automated detection of style issues (see ament_lint_auto). [待校准@8065]

C [待校准@8066]

Standard [待校准@8067]

We will target C99. [待校准@8068]

Style [待校准@8069]

We will use Python's PEP7 for our C style guide, with some modifications and additions: [待校准@8070]

  • We will target C99, as we do not need to support C89 (as PEP7 recommends) [待校准@8071]

    • rationale: among other things it allows us to use both // and /* */ style comments [待校准@8072]

    • rationale: C99 is pretty much ubiquitous now [待校准@8073]

  • C++ style // comments are allowed [待校准@8074]

  • (optional) Always place literals on the left-hand side of comparison operators, e.g. 0 == ret instead of ret == 0 [待校准@8075]

    • rationale: ret == 0 too easily turns into ret = 0 by accident [待校准@8076]

    • optional because when using -Wall (or equivalent) modern compilers will warn you when this happens [待校准@8077]

All of the following modifications only apply if we are not writing Python modules: [待校准@8078]

  • Do not use Py_ as a prefix for everything [待校准@8079]

    • instead use a CamelCase version of the package name or other appropriate prefix [待校准@8080]

  • The stuff about documentation strings doesn't apply [待校准@8081]

We can use the pep7 python module for style checking. The editor integration seems slim, we may need to look into automated checking for C in more detail. [待校准@8082]

C++ [待校准@8083]

Standard [待校准@8067]

Humble targets C++17. [待校准@8084]

Style [待校准@8069]

We will use the Google C++ Style Guide, with some modifications: [待校准@8085]

Line Length [待校准@8086]

  • Our maximum line length is 100 characters. [待校准@8087]

File Extensions [待校准@8088]

  • Header files should use the .hpp extension. [待校准@8089]

    • rationale: Allow tools to determine content of files, C++ or C. [待校准@8090]

  • Implementation files should use the .cpp extension. [待校准@8091]

    • rationale: Allow tools to determine content of files, C++ or C. [待校准@8090]

Variable Naming [待校准@8092]

  • For global variables use lowercase with underscores prefixed with g_ [待校准@8093]

    • rationale: keep variable naming case consistent across the project [待校准@8094]

    • rationale: easy to tell the scope of a variable at a glance [待校准@8095]

    • consistency across languages [待校准@8096]

Function and Method Naming [待校准@8097]

  • Google style guide says CamelCase, but the C++ std library's style of snake_case is also allowed [待校准@8098]

    • rationale: ROS 2 core packages currently use snake_case [待校准@8099]

      • reason: either an historical oversight or a personal preference that didn't get checked by the linter [待校准@8100]

      • reason for not changing: retroactively changing would be too disruptive [待校准@8101]

    • other considerations: [待校准@8102]

      • cpplint.py does not check this case (hard to enforce other than with review) [待校准@8103]

      • snake_case can result in more consistency across languages [待校准@8104]

    • specific guidance: [待校准@8105]

      • for existing projects, prefer the existing style [待校准@8106]

      • for new projects, either is acceptable, but a preference for matching related existing projects is advised [待校准@8107]

      • final decision is always developer discretion [待校准@8108]

        • special cases like function pointers, callable types, etc. may require bending the rules [待校准@8109]

      • Note that classes should still use CamelCase by default [待校准@8110]

Access Control [待校准@8111]

  • Drop requirement for all class members to be private and therefore require accessors [待校准@8112]

    • rationale: this is overly constraining for user API design [待校准@8113]

    • we should prefer private members, only making them public when they are needed [待校准@8114]

    • we should consider using accessors before choosing to allow direct member access [待校准@8115]

    • we should have a good reason for allowing direct member access, other than because it is convenient for us [待校准@8116]

Exceptions [待校准@8117]

  • Exceptions are allowed [待校准@8118]

    • rationale: this is a new codebase, so the legacy argument doesn't apply to us [待校准@8119]

    • rationale: for user-facing API's it is more idiomatic C++ to have exceptions [待校准@8120]

    • Exceptions in destructors should be explicitly avoided [待校准@8121]

  • We should consider avoiding Exceptions if we intend to wrap the resulting API in C [待校准@8122]

    • rationale: it will make it easier to wrap in C [待校准@8123]

    • rationale: most of our dependencies in code we intend to wrap in C do not use exceptions anyways [待校准@8124]

Function-like Objects [待校准@8125]

  • No restrictions on Lambda's or std::function or std::bind [待校准@8126]

Boost [待校准@8127]

  • Boost should be avoided unless absolutely required. [待校准@8128]

Comments and Doc Comments [待校准@8129]

  • Use /// and /** */ comments for documentation purposes and // style comments for notes and general comments [待校准@8130]

    • Class and Function comments should use /// and /** */ style comments [待校准@8131]

    • rationale: these are recommended for Doxygen and Sphinx in C/C++ [待校准@8132]

    • rationale: mixing /* */ and // is convenient for block commenting out code which contains comments [待校准@8133]

    • Descriptions of how the code works or notes within classes and functions should use // style comments [待校准@8134]

Pointer Syntax Alignment [待校准@8135]

  • Use char * c; instead of char* c; or char *c; because of this scenario char* c, *d, *e; [待校准@8136]

Class Privacy Keywords [待校准@8137]

  • Do not put 1 space before public:, private:, or protected:, it is more consistent for all indentions to be a multiple of 2 [待校准@8138]

    • rationale: most editors don't like indentions which are not a multiple of the (soft) tab size [待校准@8139]

    • Use zero spaces before public:, private:, or protected:, or 2 spaces [待校准@8140]

    • If you use 2 spaces before, indent other class statements by 2 additional spaces [待校准@8141]

    • Prefer zero spaces, i.e. public:, private:, or protected: in the same column as the class [待校准@8142]

Nested Templates [待校准@8143]

  • Never add whitespace to nested templates [待校准@8144]

    • Prefer set<list<string>> (C++11 feature) to set<list<string> > or set< list<string> > [待校准@8145]

Always Use Braces [待校准@8146]

  • Always use braces following if, else, do, while, and for, even when the body is a single line. [待校准@8147]

    • rationale: less opportunity for visual ambiguity and for complications due to use of macros in the body [待校准@8148]

Open Versus Cuddled Braces [待校准@8149]

  • Use open braces for function, class, enum, and struct definitions, but cuddle braces on if, else, while, for, etc... [待校准@8150]

    • Exception: when an if (or while, etc.) condition is long enough to require line-wrapping, then use an open brace (i.e., don't cuddle). [待校准@8151]

  • When a function call cannot fit on one line, wrap at the open parenthesis (not in between arguments) and start them on the next line with a 2-space indent. Continue with the 2-space indent on subsequent lines for more arguments. (Note that the Google style guide is internally contradictory on this point.) [待校准@8152]

    • Same goes for if (and while, etc.) conditions that are too long to fit on one line. [待校准@8153]

Examples [待校准@8154]

This is OK: [待校准@8155]

int main(int argc, char **argv)
{
  if (condition) {
    return 0;
  } else {
    return 1;
  }
}

if (this && that || both) {
  ...
}

// Long condition; open brace
if (
  this && that || both && this && that || both && this && that || both && this && that)
{
  ...
}

// Short function call
call_func(foo, bar);

// Long function call; wrap at the open parenthesis
call_func(
  foo, bar, foo, bar, foo, bar, foo, bar, foo, bar, foo, bar, foo, bar, foo, bar, foo, bar,

Python [待校准@8169]

Style [待校准@8069]

We will use the PEP8 guidelines for code format. [待校准@8172]

We chose the following more precise rule where PEP 8 leaves some freedom: [待校准@8173]

  • We allow up to 100 characters per line (fifth paragraph). [待校准@8174]

  • We pick single quotes over double quotes as long as no escaping is necessary. [待校准@8175]

  • We prefer hanging indents for continuation lines. [待校准@8176]

Tools like the (ament_)pycodestyle Python package should be used in unit-test and/or editor integration for checking Python code style. [待校准@8177]

CMake [待校准@8184]

Version [待校准@8170]

We will target CMake 3.8. [待校准@8185]

Style [待校准@8069]

Since there is not an existing CMake style guide we will define our own: [待校准@8186]

  • Use lowercase command names (find_package, not FIND_PACKAGE). [待校准@8187]

  • Use snake_case identifiers (variables, functions, macros). [待校准@8188]

  • Use empty else() and end...() commands. [待校准@8189]

  • No whitespace before ('s. [待校准@8190]

  • Use two spaces of indention, do not use tabs. [待校准@8191]

  • Do not use aligned indentation for parameters of multi-line macro invocations. Use two spaces only. [待校准@8192]

  • Prefer functions with set(PARENT_SCOPE) to macros. [待校准@8193]

  • When using macros prefix local variables with _ or a reasonable prefix. [待校准@8194]

Markdown / reStructured Text / docblocks [待校准@8195]

Style [待校准@8069]

The following rules to format text is intended to increase readability as well as versioning. [待校准@8196]

  • [.md, .rst only] Each section title should be preceded by one empty line and succeeded by one empty line. [待校准@8197]

    • Rationale: It expedites to get an overview about the structure when screening the document. [待校准@8198]

  • [.rst only] In reStructured Text the headings should follow the hierarchy described in the Sphinx style guide: [待校准@8199]

    • # with overline (only once, used for the document title) [待校准@8200]

    • * with overline [待校准@8201]

    • = [待校准@8202]

    • - [待校准@8203]

    • ^ [待校准@8204]

    • " [待校准@8205]

    • Rationale: A consistent hierarchy expedites getting an idea about the nesting level when screening the document. [待校准@8206]

  • [.md only] In Markdown the headings should follow the ATX-style described in the Markdown syntax documentation [待校准@8207]

    • ATX-style headers use 1-6 hash characters (#) at the start of the line to denote header levels 1-6. [待校准@8208]

    • A space between the hashes and the header title should be used (such as # Heading 1) to make it easier to visually separate them. [待校准@8209]

    • Justification for the ATX-style preference comes from the Google Markdown style guide [待校准@8210]

    • Rationale: ATX-style headers are easier to search and maintain, and make the first two header levels consistent with the other levels. [待校准@8211]

  • [any] Each sentence must start on a new line. [待校准@8212]

    • Rationale: For longer paragraphs a single change in the beginning makes the diff unreadable since it carries forward through the whole paragraph. [待校准@8213]

  • [any] Each sentence can optionally be wrapped to keep each line short. [待校准@8214]

  • [any] The lines should not have any trailing white spaces. [待校准@8215]

  • [.md, .rst only] A code block must be preceded and succeeded by an empty line. [待校准@8216]

    • Rationale: Whitespace is significant only directly before and directly after fenced code blocks. Following these instructions will ensure that highlighting works properly and consistently. [待校准@8217]

  • [.md, .rst only] A code block should specify a syntax (e.g. bash). [待校准@8218]
