This page defines the practices and policies we employ when developing ROS 2. [待校准@1645]

General Principles [待校准@1646]

Some principles are common to all ROS 2 development: [待校准@1647]

  • Shared ownership: Everybody working on ROS 2 should feel ownership over all parts of the system. The original author of a chunk of code does not have any special permission or obligation to control or maintain that chunk of code. Everyone is free to propose changes anywhere, to handle any type of ticket, and to review any pull request. [待校准@1648]

  • Be willing to work on anything: As a corollary to shared ownership, everybody should be willing to take on any available task and contribute to any aspect of the system. [待校准@1649]

  • Ask for help: If you run into trouble on something, ask your fellow developers for help, via tickets, comments, or email, as appropriate. [待校准@1650]

Quality Practices [待校准@1651]

Packages can ascribe to different levels of quality based on the development practices they adhere to, as per the guidelines in REP 2004: Package Quality Categories. The categories are differentiated by their policies on versioning, testing, documentation, and more. [待校准@1652]

The following sections are the specific development rules we follow to ensure core packages are of the highest quality ('Level 1'). We recommend all ROS developers strive to adhere to the following policies to ensure quality across the ROS ecosystem. [待校准@1653]

Versioning [待校准@1654]

We will use the Semantic Versioning guidelines (semver) for versioning. [待校准@1655]

We will also adhere to some ROS-specific rules built on top of semver's full meaning: [待校准@1656]

  • Major version increments (i.e. breaking changes) should not be made within a released ROS distribution. [待校准@1657]

    • Patch (interface-preserving) and minor (non-breaking) version increments do not break compatibility, so these sorts of changes are allowed within a release. [待校准@1658]

    • Major ROS releases are the best time to release breaking changes. If a core package needs multiple breaking changes, they should be merged into their integration branch (e.g. rolling) to allow catching problems in CI quickly, but released together to reduce the number of major releases for ROS users. [待校准@1659]

    • Though major increments require a new distribution, a new distribution does not necessarily require a major bump (if development and release can happen without breaking API). [待校准@1660]

  • For compiled code, the ABI is considered part of the public interface. Any change that requires recompiling dependent code is considered major (breaking). [待校准@1661]

    • ABI breaking changes can be made in a minor version bump before a distribution release (getting added to the rolling release). [待校准@1662]

  • We enforce API stability for core packages in Dashing and Eloquent even though their major version components are 0, despite SemVer's specification regarding initial development. [待校准@1663]

    • Subsequently, packages should strive to reach a mature state and increase to version 1.0.0 so to match semver's specifications. [待校准@1664]

Caveats [待校准@1665]

These rules are best-effort. In unlikely, extreme cases, it may be necessary to break API within a major version/distribution. Whether an unplanned break increments the major or minor version will be assessed on a case-by-case basis. [待校准@1666]

For example, consider a situation involving released X-turtle, corresponding to major version 1.0.0, and released Y-turtle, corresponding to major version 2.0.0. [待校准@1667]

If an API-breaking fix is identified to be absolutely necessary in X-turtle, bumping to 2.0.0 is obviously not an option because 2.0.0 already exists. [待校准@1668]

The solutions for handling X-turtle's version in such a case, both non-ideal, are: [待校准@1669]

  1. Bumping X-turtle's minor version: non-ideal because it violates SemVer's principle that breaking changes must bump the major version. [待校准@1670]

  2. Bumping X-turtle's major version past Y-turtle (to 3.0.0): non-ideal because the older distro's version would become higher than the already-available version of a newer distro, which would invalidate/break version-specific conditional code. [待校准@1671]

The developer will have to decide which solution to use, or more importantly, which principle they are willing to break. We cannot suggest one or the other, but in either case we do require that explicit measures be taken to communicate the disruption and its explanation to users manually (beyond just the version increment). [待校准@1672]

If there were no Y-turtle, even though the fix would technically just be a patch, X-turtle would have to bump to 2.0.0. This case adheres to SemVer, but breaks from our own rule that major increments should not be introduced in a released distribution. [待校准@1673]

This is why we consider the versioning rules best-effort. As unlikely as the examples above are, it is important to accurately define our versioning system. [待校准@1674]

Public API declaration [待校准@1675]

According to semver, every package must clearly declare a public API. We will use the "Public API Declaration" section of the quality declaration of a package to declare what symbols are part of the public API. [待校准@1676]

For most C and C++ packages the declaration is any header that it installs. However, it is acceptable to define a set of symbols which are considered private. Avoiding private symbols in headers can help with ABI stability, but is not required. [待校准@1677]

For other languages like Python, a public API must be explicitly defined, so that it is clear what symbols can be relied on with respect to the versioning guidelines. The public API can also be extended to build artifacts like configuration variables, CMake config files, etc. as well as executables and command-line options and output. Any elements of the public API should be clearly stated in the package's documentation. If something you are using is not explicitly listed as part of the public API in the package's documentation, then you cannot depend on it not changing between minor or patch versions. [待校准@1678]

Deprecation strategy [待校准@1679]

Where possible, we will also use the tick-tock deprecation and migration strategy for major version increments. New deprecations will come in a new distribution release, accompanied by compiler warnings expressing that the functionality is being deprecated. In the next release, the functionality will be completely removed (no warnings). [待校准@1680]

Example of function foo deprecated and replaced by function bar: [待校准@1681]

Version [待校准@1523]

API [待校准@1682]

X-turtle [待校准@1683]

void foo(); [待校准@1684]

Y-turtle [待校准@1685]

[[deprecated("use bar()")]] void foo(); <br> void bar(); [待校准@1686]

Z-turtle [待校准@1687]

void bar(); [待校准@1688]

We must not add deprecations after a distribution is released. Deprecations do not necessarily require a major version bump, though. A deprecation can be introduced in a minor version bump if the bump happens before the distro is released (similar to ABI breaking changes). [待校准@1689]

For example, if X-turtle begins development as 2.0.0, a deprecation can be added in 2.1.0 before X-turtle is released. [待校准@1690]

We will attempt to maintain compatibility across distros as much as possible. However, like the caveats associated with SemVer, tick-tock or even deprecation in general may be impossible to completely adhere to in certain cases. [待校准@1691]

Change control process [待校准@1692]

  • All changes must go through a pull request. [待校准@1693]

  • We will enforce the Developer Certificate of Origin (DCO) on pull requests in ROSCore repositories. [待校准@1694]

    • It requires all commit messages to contain the Signed-off-by line with an email address that matches the commit author. [待校准@1695]

    • You can pass -s / --signoff to the git commit invocation or write the expected message manually (e.g. Signed-off-by: Your Name Developer <your.name@example.com>). [待校准@1696]

    • DCO is not required for pull requests that only address whitespace removal, typo correction, and other trivial changes. [待校准@1697]

  • Always run CI jobs for all tier 1 platforms for every pull request and include links to jobs in the pull request. (If you don't have access to the Jenkins jobs someone will trigger the jobs for you.) [待校准@1698]

  • A minimum of 1 approval from a fellow developer who did not author the pull request is required to consider it approved. Approval is required before merging. [待校准@1699]

    • Packages may choose to increase this number. [待校准@1700]

  • Any required changes to documentation (API documentation, feature documentation, release notes, etc.) must be proposed before merging related changes. [待校准@1701]

Guidelines for backporting PRs [待校准@1702]

When changing an older version of ROS: [待校准@1703]

  • Make sure the features or fixes are accepted and merged in the rolling branch before opening a PR to backport the changes to older versions. [待校准@1704]

  • When backporting to older versions, also consider backporting to any other still supported versions, even non-LTS versions. [待校准@1705]

  • If you are backporting a single PR in its entirety, title the backport PR "[Distro] <name of original PR>". If backporting a subset of changes from one or multiple PRs, the title should be "[Distro] <description of changes>". [待校准@1706]

  • Link to all PRs whose changes you're backporting from the description of your backport PR. In a Dashing backport of a Foxy change, you do not need to link to the Eloquent backport of the same change. [待校准@1707]

Documentation [待校准@1708]

All packages should have these documentation elements present in their README or linked to from their README: [待校准@1709]

  • Description and purpose [待校准@1710]

  • Definition and description of the public API [待校准@1711]

  • Examples [待校准@1507]

  • How to build and install (should reference external tools/workflows) [待校准@1712]

  • How to build and run tests [待校准@1713]

  • How to build documentation [待校准@1714]

  • How to develop (useful for describing things like python setup.py develop) [待校准@1715]

  • License and copyright statements [待校准@1716]

Each source file must have a license and copyright statement, checked with an automated linter. [待校准@1717]

Each package must have a LICENSE file, typically the Apache 2.0 license, unless the package has an existing permissive license (e.g. rviz uses three-clause BSD). [待校准@1718]

Each package should describe itself and its purpose assuming, as much as possible, that the reader has stumbled onto it without previous knowledge of ROS or other related projects. [待校准@1719]

Each package should define and describe its public API so that there is a reasonable expectation for users about what is covered by the semantic versioning policy. Even in C and C++, where the public API can be enforced by API and ABI checking, it is a good opportunity to describe the layout of the code and the function of each part of the code. [待校准@1720]

It should be easy to take any package and from that package's documentation understand how to build, run, build and run tests, and build the documentation. Obviously we should avoid repeating ourselves for common workflows, like building a package in a workspace, but the basic workflows should be either described or referenced. [待校准@1721]

Finally, it should include any documentation for developers. This might include workflows for testing the code using something like python setup.py develop, or it might mean describing how to make use of extension points provided by your package. [待校准@1722]

Examples: [待校准@1723]

  • capabilities: https://docs.ros.org/hydro/api/capabilities/html/ [待校准@1724]

    • This one gives an example of docs which describe the public API [待校准@1725]

  • catkin_tools: https://catkin-tools.readthedocs.org/en/latest/development/extending_the_catkin_command.html [待校准@1726]

    • This is an example of describing an extension point for a package [待校准@1727]

(API docs are not yet being automatically generated) [待校准@1728]

Testing [待校准@1729]

All packages should have some level of system, integration, and/or unit tests. [待校准@1730]

Unit tests should always be in the package which is being tested and should make use of tools like Mock to try and test narrow parts of the codebase in constructed scenarios. Unit tests should not bring in test dependencies that are not testing tools, e.g. gtest, nosetest, pytest, mock, etc... [待校准@1731]

Integration tests can test interactions between parts of the code or between parts of the code and the system. They often test software interfaces in ways that we expect the user to use them. Like Unit tests, Integration tests should be in the package which is being tested and should not bring in non-tool test dependencies unless absolutely necessary, i.e. all non-tool dependencies should only be allowed under extreme scrutiny so they should be avoided if possible. [待校准@1732]

System tests are designed to test end-to-end situations between packages and should be in their own packages to avoid bloating or coupling packages and to avoid circular dependencies. [待校准@1733]

In general external or cross package test dependencies should be minimized to prevent circular dependencies and tightly coupled test packages. [待校准@1734]

All packages should have some unit tests and possibly integration tests, but the degree to which they should have them is based on the package's quality category. The following subsections apply to 'Level 1' packages: [待校准@1735]

Code coverage [待校准@1736]

We will provide line coverage, and achieve line coverage above 95%. If a lower percentage target is justifiable, it must be prominently documented. We may provide branch coverage, or exclude code from coverage (test code, debug code, etc.). We require that coverage increase or stay the same before merging a change, but it may be acceptable to make a change that decreases code coverage with proper justification (e.g. deleting code that was previously covered can cause the percentage to drop). [待校准@1737]

Performance [待校准@1738]

We strongly recommend performance tests, but recognize they don't make sense for some packages. If there are performance tests, we will choose to either check each change or before each release or both. We will also require justification for merging a change or making a release that lowers performance. [待校准@1739]

Linters and static analysis [待校准@1740]

We will use ROS code style and enforce it with linters from ament_lint_common. All linters/static analysis that are part of ament_lint_common must be used. [待校准@1741]

The ament_lint_auto documentation provides information on running ament_lint_common. [待校准@1742]

General Practices [待校准@1743]

Some practices are common to all ROS 2 development. [待校准@1744]

These practices don't affect package quality level as described in REP 2004, but are still highly recommended for the development process. [待校准@1745]

Issues [待校准@1746]

When filing an issue please make sure to: [待校准@1747]

  • Include enough information for another person to understand the issue. In ROS 2, the following points are needed for narrowing down the cause of an issue. Testing with as many alternatives in each category as feasible will be especially helpful. [待校准@1748]

    • The operating system and version. Reasoning: ROS 2 supports multiple platforms, and some bugs are specific to particular versions of operating systems/compilers. [待校准@1749]

    • The installation method. Reasoning: Some issues only manifest if ROS 2 has been installed from "fat archives" or from Debians. This can help us determine if the issue is with the packaging process. [待校准@1750]

    • The specific version of ROS 2. Reasoning: Some bugs may be present in a particular ROS 2 release and later fixed. It is important to know if your installation includes these fixes. [待校准@1751]

    • The DDS/RMW implementation being used (see this page for how to determine which one). Reasoning: Communication issues may be specific to the underlying ROS middleware being used. [待校准@1752]

    • The ROS 2 client library being used. Reasoning: This helps us narrow down the layer in the stack at which the issue might be. [待校准@1753]

  • Include a list of steps to reproduce the issue. [待校准@1754]

  • In case of a bug consider to provide a short, self contained, correct (compilable), example. Issues are much more likely to be resolved if others can reproduce them easily. [待校准@1755]

  • Mention troubleshooting steps that have been tried already, including: [待校准@1756]

    • Upgrading to the latest version of the code, which may include bug fixes that have not been released yet. See this section and follow the instructions to get the "rolling" branches. [待校准@1757]

    • Trying with a different RMW implementation. See this page for how to do that. [待校准@1758]

Branches [待校准@1759]

注解

These are just guidelines. It is up to the package maintainer to choose branch names that match their own workflow. [待校准@1760]

It is good practice to have separate branches in a package's source repository for each ROS distribution it is targeting. These branches are typically named after the distribution they target. For example, a humble branch for development targeted specifically at the Humble distribution. [待校准@1761]

Releases are also made from these branches, targeting the appropriate distribution. Development targeted at a specific ROS distribution can happen on the appropriate branch. For example: Development commits targeting foxy are made to the foxy branch, and package releases for foxy are made from that same branch. [待校准@1762]

注解

This requires the package maintainers to perform backports or forwardports as appropriate to keep all branches up to date with features. The maintainers must also perform general maintenance (bug fixes, etc.) on all branches from which package releases are still made. [待校准@1763]

For example, if a feature is merged into the Rolling-specific branch (e.g. rolling or main), and that feature is also appropriate to the Humble distribution (does not break API, etc.), then it is good practice to backport the feature to the Humble-specific branch. [待校准@1764]

The maintainers may make releases for those older distributions if there are new features or bug fixes available. [待校准@1765]

What about main and rolling ? [待校准@1766]

main typically targets Rolling (and so, the next unreleased ROS distribution), though the maintainers may decide to develop and release from a rolling branch instead. [待校准@1767]

Pull requests [待校准@1768]

  • A pull request should only focus on one change. Separate changes should go into separate pull requests. See GitHub's guide to writing the perfect pull request. [待校准@1769]

  • A patch should be minimal in size and avoid any kind of unnecessary changes. [待校准@1770]

  • A pull request must contain minimum number of meaningful commits. [待校准@1771]

    • You can create new commits while the pull request is under review. [待校准@1772]

  • Before merging a pull request all changes should be squashed into a small number of semantic commits to keep the history clear. [待校准@1773]

    • But avoid squashing commits while a pull request is under review. Your reviewers might not notice that you made the change, thereby introducing potential for confusion. Plus, you're going to squash before merging anyway; there's no benefit to doing it early. [待校准@1774]

  • Any developer is welcome to review and approve a pull request (see General Principles). [待校准@1775]

  • When you are working on a change that is not ready for review or to be merged, use a draft pull request. When that change is ready for review, move the pull request out of the draft state. Note that if you want early feedback from specific people on a draft pull request, you can @ mention them in the pull request's description or in a comment on the pull request. [待校准@1776]

  • If your pull request depends on other pull requests, link to each depended on pull request by adding - Depends on <link> at the top of your pull request's description. Doing so helps reviewers understand the context of the pull request. [待校准@1777]

  • When you start reviewing a pull request, comment on the pull request so that other developers know that you're reviewing it. [待校准@1778]

  • Pull-request review is not read-only, with the reviewer making comments and then waiting for the author to address them. As a reviewer, feel free to make minor improvements (typos, style issues, etc.) in-place. As the opener of a pull-request, if you are working in a fork, checking the box to allow edits from upstream contributors will assist with the aforementioned. As a reviewer, also feel free to make more substantial improvements, but consider putting them in a separate branch (either mention the new branch in a comment, or open another pull request from the new branch to the original branch). [待校准@1779]

  • Any developer (the author, the reviewer, or somebody else) can merge any approved pull request. [待校准@1780]

Library versioning [待校准@1781]

We will version all libraries within a package together. This means that libraries inherit their version from the package. This keeps library and package versions from diverging and shares reasoning with the policy of releasing packages which share a repository together. If you need libraries to have different versions then consider splitting them into different packages. [待校准@1782]

Development process [待校准@1783]

  • The default branch (in most cases the rolling branch) must always build, pass all tests and compile without warnings. If at any time there is a regression it is the top priority to restore at least the previous state. [待校准@1784]

  • Always build with tests enabled. [待校准@1785]

  • Always run tests locally after changes and before proposing them in a pull request. Besides using automated tests, also run the modified code path manually to ensure that the patch works as intended. [待校准@1786]

  • Always run CI jobs for all platforms for every pull request and include links to the jobs in the pull request. [待校准@1787]

For more details on recommended software development workflow, see Software Development Lifecycle section. [待校准@1788]

Changes to RMW API [待校准@1789]

When updating RMW API, it is required that RMW implementations for the Tier 1 middleware libraries are updated as well. For example, a new function rmw_foo() introduced to the RMW API must be implemented in the following packages (as of ROS Galactic): [待校准@1790]

  • rmw_connextdds [待校准@1791]

  • rmw_cyclonedds [待校准@1792]

  • rmw_fastrtps [待校准@1793]

Updates for non-Tier 1 middleware libraries should also be considered if feasible (e.g. depending on the size of the change). See REP-2000 for the list of middleware libraries and their tiers. [待校准@1794]

Tracking tasks [待校准@1795]

To help organize work on ROS 2, the core ROS 2 development team uses kanban-style GitHub project boards. [待校准@1796]

Not all issues and pull requests are tracked on the project boards, however. A board usually represents an upcoming release or specific project. Tickets can be browsed on a per-repo basis by browsing the ROS 2 repositories' individual issue pages. [待校准@1797]

The names and purposes of columns in any given ROS 2 project board vary, but typically follow the same general structure: [待校准@1798]

  • To do: Issues that are relevant to the project, ready to be assigned [待校准@1799]

  • In progress: Active pull requests on which work is currently in progress [待校准@1800]

  • In review: Pull requests where work is complete and ready for review, and for those currently under active review [待校准@1801]

  • Done: Pull requests and related issues are merged/closed (for informational purposes) [待校准@1802]

To request permission to make changes, simply comment on the tickets you're interested in. Depending on the complexity, it might be useful to describe how you plan to address it. We will update the status (if you don't have the permission) and you can start working on a pull request. If you contribute regularly we will likely just grant you permission to manage the labels etc. yourself. [待校准@1803]

Programming conventions [待校准@1804]

  • Defensive programming: ensure that assumptions are held as early as possible. E.g. check every return code and make sure to at least throw an exception until the case is handled more gracefully. [待校准@1805]

  • All error messages must be directed to stderr. [待校准@1806]

  • Declare variables in the narrowest scope possible. [待校准@1807]

  • Keep group of items (dependencies, imports, includes, etc.) ordered alphabetically. [待校准@1808]

C++ specific [待校准@1809]

  • Avoid using direct streaming (<<) to stdout / stderr to prevent interleaving between multiple threads. [待校准@1810]

  • Avoid using references for std::shared_ptr since that subverts the reference counting. If the original instance goes out of scope and the reference is being used it accesses freed memory. [待校准@1811]

Filesystem layout [待校准@1812]

The filesystem layout of packages and repositories should follow the same conventions in order to provide a consistent experience for users browsing our source code. [待校准@1813]

Package layout [待校准@1814]

  • src: contains all C and C++ code [待校准@1815]

    • Also contains C/C++ headers which are not installed [待校准@1816]

  • include: contains all C and C++ headers which are installed [待校准@1817]

    • <package name>: for all C and C++ installed headers they should be folder namespaced by the package name [待校准@1818]

  • <package_name>: contains all Python code [待校准@1819]

  • test: contains all automated tests and test data [待校准@1820]

  • config: contains configuration files, e.g. YAML parameters files and RViz config files [待校准@1821]

  • doc: contains all the documentation [待校准@1822]

  • launch: contains all launch files [待校准@1823]

  • package.xml: as defined by REP-0140 (may be updated for prototyping) [待校准@1824]

  • CMakeLists.txt: only ROS packages which use CMake [待校准@1825]

  • setup.py: only ROS packages which use Python code only [待校准@1826]

  • README: can be rendered on GitHub as a landing page for the project [待校准@1827]

    • This can be as short or detailed as is convenient, but it should at least link to project documentation [待校准@1828]

    • Consider putting a CI or code coverage tag in this README [待校准@1829]

    • It can also be .rst or anything else that GitHub supports [待校准@1830]

  • CONTRIBUTING: describes the contribution guidelines [待校准@1831]

    • This might include license implication, e.g. when using the Apache 2 License. [待校准@1832]

  • LICENSE: a copy of the license or licenses for this package [待校准@1833]

  • CHANGELOG.rst: REP-0132 compliant changelog [待校准@1834]

Repository layout [待校准@1835]

Each package should be in a subfolder which has the same name as the package. If a repository contains only a single package it can optionally be in the root of the repository. [待校准@1836]

Developer Workflow [待校准@1837]

We track open tickets and active PRs related to upcoming releases and larger projects using GitHub project boards. [待校准@1838]

The usual workflow is: [待校准@1839]

  • Discuss design (GitHub ticket on the appropriate repository, and a design PR to https://github.com/ros2/design if needed) [待校准@1840]

  • Write implementation on a feature branch on a fork [待校准@1841]

    • Please check out the developer guide for guidelines and best practices [待校准@1842]

  • Write tests [待校准@1843]

  • Enable and run linters [待校准@1844]

  • Run tests locally using colcon test (see the colcon tutorial) [待校准@1845]

  • Once everything builds locally without warnings and all tests are passing, run CI on your feature branch: [待校准@1846]

    • Go to ci.ros2.org [待校准@1847]

    • Log in (top right corner) [待校准@1848]

    • Click on the ci_launcher job [待校准@1849]

    • Click "Build with Parameters" (left column) [待校准@1850]

    • In the first box "CI_BRANCH_TO_TEST" enter your feature branch name [待校准@1851]

    • Hit the build button [待校准@1852]

    (if you are not a ROS 2 committer, you don't have access to the CI farm. In that case, ping the reviewer of your PR to run CI for you) [待校准@1853]

  • If your use case requires running code coverage: [待校准@1854]

    • Go to ci.ros2.org [待校准@1847]

    • Log in (top right corner) [待校准@1848]

    • Click on the ci_linux_coverage job [待校准@1855]

    • Click "Build with Parameters" (left column) [待校准@1850]

    • Be sure of leaving "CI_BUILD_ARGS" and "CI_TEST_ARGS" with the default values [待校准@1856]

    • Hit the build button [待校准@1852]

    • At the end of the document there are instructions on how to interpret the result of the report and calculate the coverage rate [待校准@1857]

  • If the CI job built without warnings, errors and test failures, post the links of your jobs on your PR or high-level ticket aggregating all your PRs (see example here) [待校准@1858]

    • Note that the markdown for these badges is in the console output of the ci_launcher job [待校准@1859]

  • When the PR has been approved: [待校准@1860]

    • the person who submitted the PR merges it using "Squash and Merge" option so that we keep a clean history [待校准@1861]

      • If the commits deserve to keep separated: squash all the nitpick/linters/typo ones together and merge the remaining set [待校准@1862]

        • Note: each PR should target a specific feature so Squash and Merge should make sense 99% of the time [待校准@1863]

  • Delete the branch once merged [待校准@1864]

Architectural Development Practices [待校准@1865]

This section describes the ideal lifecycle that should be employed when making large architectural changes to ROS 2. [待校准@1866]

Software Development Lifecycle [待校准@1867]

This section describes step-by-step how to plan, design, and implement a new feature: [待校准@1868]

  1. Task Creation [待校准@1869]

  2. Creating the Design Document [待校准@1870]

  3. Design Review [待校准@1871]

  4. Implementation [待校准@1872]

  5. Code Review [待校准@1873]

Task creation [待校准@1874]

Tasks requiring changes to critical parts of ROS 2 should have design reviews during early stages of the release cycle. If a design review is happening in the later stages, the changes will be part of a future release. [待校准@1875]

  • An issue should be created in the appropriate ros2 repository, clearly describing the task being worked on. [待校准@1876]

    • It should have a clear success criteria and highlight the concrete improvements expected from it. [待校准@1877]

    • If the feature is targeting a ROS release, ensure this is tracked in the ROS release ticket (example). [待校准@1878]

Writing the design document [待校准@1879]

Design docs must never include confidential information. Whether or not a design document is required for your change depends on how big the task is. [待校准@1880]

  1. You are making a small change or fixing a bug: [待校准@1881]

  • A design document is not required, but an issue should be opened in the appropriate repository to track the work and avoid duplication of efforts. [待校准@1882]

  1. You are implementing a new feature or would like to contribute to OSRF-owned infrastructure (like Jenkins CI): [待校准@1883]

  • Design doc is required and should be contributed to ros2/design to be made accessible on https://design.ros2.org/. [待校准@1884]

  • You should fork the repository and submit a pull request detailing the design. [待校准@1885]

Mention the related ros2 issue (for example, Design doc for task ros2/ros2#<issue id>) in the pull request or the commit message. Detailed instructions are on the ROS 2 Contribute page. Design comments will be made directly on the pull request. [待校准@1886]

If the task is planned to be released with a specific version of ROS, this information should be included in the pull request. [待校准@1887]

Design document review [待校准@1888]

Once the design is ready for review, a pull request should be opened and appropriate reviewers should be assigned. It is recommended to include project owner(s) - maintainers of all impacted packages (as defined by package.xml maintainer field, see REP-140) - as reviewers. [待校准@1889]

  • If the design doc is complex or reviewers have conflicting schedules, an optional design review meeting can be set up. In this case, [待校准@1890]

    Before the meeting [待校准@1891]

    • Send a meeting invite at least one week in advance [待校准@1892]

    • Meeting duration of one hour is recommended [待校准@1893]

    • Meeting invite should list all decisions to be made during the review (decisions requiring package maintainer approval) [待校准@1894]

    • Meeting required attendees: design pull request reviewers [待校准@1895]

      Meeting optional attendees: all OSRF engineers, if applicable [待校准@1896]

    During the meeting [待校准@1897]

    • The task owner drives the meeting, presents their ideas and manages discussions to ensure an agreement is reached on time [待校准@1898]

    After the meeting [待校准@1899]

    • The task owner should send back meeting notes to all attendees [待校准@1900]

    • If minor issues have been raised about the design: [待校准@1901]

      • The task owner should update the design doc pull request based on the feedback [待校准@1902]

      • Additional review is not required [待校准@1903]

    • If major issues have been raised about the design: [待校准@1904]

      • It is acceptable to remove sections for which there is no clear agreement [待校准@1905]

      • The debatable parts of the design can be resubmitted as a separate task in the future [待校准@1906]

      • If removing the debatable parts is not an option, work directly with package owners to reach an agreement [待校准@1907]

  • Once consensus is reached: [待校准@1908]

    • Ensure the ros2/design pull request has been merged, if applicable [待校准@1909]

    • Update and close the GitHub issue associated with this design task [待校准@1910]

Implementation [待校准@1872]

Before starting, go through the Pull requests section for best practices. [待校准@1911]

  • For each repo to be modified: [待校准@1912]

    • Modify the code, go to the next step if finished or at regular intervals to backup your work. [待校准@1913]

    • Self-review your changes using git add -i. [待校准@1914]

    • Create a new signed commit using git commit -s. [待校准@1915]

      • A pull request should contain minimal semantically meaningful commits (for instance, a large number of 1-line commits is not acceptable). Create new fixup commits while iterating on feedback, or optionally, amend existing commits using git commit --amend if you don't want to create a new commit every time. [待校准@1916]

      • Each commit must have a properly written, meaningful, commit message. More instructions here. [待校准@1917]

      • Moving files must be done in a separate commit, otherwise git may fail to accurately track the file history. [待校准@1918]

      • Either the pull request description or the commit message must contain a reference to the related ros2 issue, so it gets automatically closed when the pull request is merged. See this doc for more details. [待校准@1919]

      • Push the new commits. [待校准@1920]

Code review [待校准@1921]

Once the change is ready for code review: [待校准@1922]

  • Open a pull request for each modified repository. [待校准@1923]

    • Remember to follow Pull requests best practices. [待校准@1924]

    • GitHub can be used to create pull requests from the command-line. [待校准@1925]

    • If the task is planned to be released with a specific version of ROS, this information should be included in each pull request. [待校准@1926]

  • Package owners who reviewed the design document should be mentioned in the pull request. [待校准@1927]

  • Code review SLO: although reviewing pull requests is best-effort, it is helpful to have reviewers comment on pull requests within a week and code authors to reply back to comments within a week, so there is no loss of context. [待校准@1928]

  • Iterate on feedback as usual, amend and update the development branch as needed. [待校准@1929]

  • Once the PR is approved, package maintainers will merge the changes in. [待校准@1930]

Build Farm Introduction [待校准@1931]

The build farm is located at ci.ros2.org. [待校准@1932]

Every night we run nightly jobs which build and run all the tests in various scenarios on various platforms. Additionally, we test all pull requests against these platforms before merging. [待校准@1933]

This is the current set of target platforms and architectures, though it evolves overtime: [待校准@1934]

  • Ubuntu 22.04 Jammy [待校准@1935]

    • amd64 [待校准@1936]

    • aarch64 [待校准@1937]

  • Windows 10 [待校准@1938]

    • amd64 [待校准@1936]

There are several categories of jobs on the buildfarm: [待校准@1939]

  • manual jobs (triggered manually by developers): [待校准@1940]

    • ci_linux: build + test the code on Ubuntu Xenial [待校准@1941]

    • ci_linux-aarch64: build + test the code on Ubuntu Xenial on an ARM 64-bit machine (aarch64) [待校准@1942]

    • ci_linux_coverage: build + test + generation of test coverage [待校准@1943]

    • ci_windows: build + test the code on Windows 10 [待校准@1944]

    • ci_launcher: trigger all the jobs listed above [待校准@1945]

  • nightly (run every night): [待校准@1946]

    • Debug: build + test the code with CMAKE_BUILD_TYPE=Debug [待校准@1947]

      • nightly_linux_debug [待校准@1948]

      • nightly_linux-aarch64_debug [待校准@1949]

      • nightly_win_deb [待校准@1950]

    • Release: build + test the code with CMAKE_BUILD_TYPE=Release [待校准@1951]

      • nightly_linux_release [待校准@1952]

      • nightly_linux-aarch64_release [待校准@1953]

      • nightly_win_rel [待校准@1954]

    • Repeated: build then run each test up to 20 times or until failed (aka flakiness hunter) [待校准@1955]

      • nightly_linux_repeated [待校准@1956]

      • nightly_linux-aarch64_repeated [待校准@1957]

      • nightly_win_rep [待校准@1958]

    • Coverage: [待校准@1959]

      • nightly_linux_coverage: build + test the code + analyses coverage for c/c++ and python [待校准@1960]

        • results are exported as a cobertura report [待校准@1961]

  • packaging (run every night; result is bundled into an archive): [待校准@1962]

    • packaging_linux [待校准@1963]

    • packaging_windows [待校准@1964]

Two additional build farms support the ROS / ROS 2 ecosystem by providing building of source and binary packages, continuous integration, testing, and analysis. [待校准@1965]

For details, frequently asked questions, and troubleshooting see build farms. [待校准@1966]

Note on Coverage runs [待校准@1967]

ROS 2 packages are organized in a way that the testing code for a given package is not only contained within the package, but could also be present in a different package. In other words: packages can exercise code belonging to other packages during the testing phase. [待校准@1968]

To achieve the coverage rate reached by all code available in the ROS 2 core packages it is recommended to run builds using a fixed set of proposed repositories. That set is defined in the default parameters of coverage jobs in Jenkins. [待校准@1969]

How to read the coverage rate from the buildfarm report [待校准@1970]

To see the coverage report for a given package: [待校准@1971]

  • When the ci_linux_coverage build finishes, click on Coverage Report [待校准@1972]

  • Scroll down to the Coverage Breakdown by Package table [待校准@1973]

  • In the table, look at the first column called "Name" [待校准@1974]

The coverage reports in the buildfarm include all the packages that were used in the ROS workspace. The coverage report includes different paths corresponding to the same package: [待校准@1975]

  • Name entries with the form: src.*.<repository_name>.<package_name>.* These correspond to the unit test runs available in a package against its own source code [待校准@1976]

  • Name entries with the form: build.<repository_name>.<package_name>.* These correspond to the unit test runs available in a package against its files generated at building or configuring time [待校准@1977]

  • Name entries with the form: install.<package_name>.* These correspond to the system/integration tests coming from testing runs of other packages [待校准@1978]

How to calculate the coverage rate from the buildfarm report [待校准@1979]

Get the combined unit coverage rate using the automatic script: [待校准@1980]

  • From the ci_linux_coverage Jenkins build copy the URL of the build [待校准@1981]

  • Download the get_coverage_ros2_pkg script [待校准@1982]

  • Execute the script: ./get_coverage_ros2_pkg.py <jenkins_build_url> <ros2_package_name> (README) [待校准@1983]

  • Grab the results from the "Combined unit testing" final line in the output of the script [待校准@1984]

Alternative: get the combined unit coverage rate from coverage report (require manual calculation): [待校准@1985]

  • When the ci_linux_coverage build finishes, click on Cobertura Coverage Report [待校准@1986]

  • Scroll down to the Coverage Breakdown by Package table [待校准@1973]

  • In the table, under the first column "Name", look for (where <package_name> is your package under testing): [待校准@1987]

    • all the directories under the pattern src.*.<repository_name>.<package_name>.* grab the two absolute values in the column "Lines". [待校准@1988]

    • all the directories under the pattern build/.<repository_name>.* grab the two absolute values in the column "Lines". [待校准@1989]

  • With the previous selection: for each cell, the first value is the lines tested and the second is the total lines of code. Aggregate all rows for getting the total of the lines tested and the total of lines of code under test. Divide to get the coverage rate. [待校准@1990]

How to measure coverage locally using lcov (Ubuntu) [待校准@1991]

To measure coverage on your own machine, install lcov. [待校准@1992]

sudo apt install -y lcov

The rest of this section assumes you are working from your colcon workspace. Compile in debug with coverage flags. Feel free to use colcon flags to target specific packages. [待校准@1993]

colcon build --cmake-args -DCMAKE_BUILD_TYPE=Debug -DCMAKE_CXX_FLAGS="${CMAKE_CXX_FLAGS} --coverage" -DCMAKE_C_FLAGS="${CMAKE_C_FLAGS} --coverage"

lcov requires an initial baseline, which you can produce with the following command. Update the output file location for your needs. [待校准@1994]

lcov --no-external --capture --initial --directory . --output-file ~/ros2_base.info

Run tests for the packages that matter for your coverage measurements. For example, if measuring rclcpp also with test_rclcpp [待校准@1995]

colcon test --packages-select rclcpp test_rclcpp

Capture the lcov results with a similar command this time dropping the --initial flag. [待校准@1996]

lcov --no-external --capture --directory . --output-file ~/ros2.info

Combine the trace .info files: [待校准@1997]

lcov --add-tracefile ~/ros2_base.info --add-tracefile ~/ros2.info --output-file ~/ros2_coverage.info

Generate html for easy visualization and annotation of covered lines. [待校准@1998]

mkdir -p coverage
genhtml ~/ros2_coverage.info --output-directory coverage
