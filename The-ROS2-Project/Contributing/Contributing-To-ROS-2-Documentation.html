Contributions to this site are most welcome. This page explains how to contribute to ROS 2 Documentation. Please be sure to read the below sections carefully before contributing. [待校准@1573]

The site is built using Sphinx, and more particularly using Sphinx multiversion. [待校准@1574]

Branch structure [待校准@1575]

The source code of documentation is located in the ROS 2 Documentation GitHub repository. This repository is set up with one branch per ROS 2 distribution to handle differences between the distributions. If a change is common to all ROS 2 distributions, it should be made to the rolling branch (and then will be backported as appropriate). If a change is specific to a particular ROS 2 distribution, it should be made to the respective branch. [待校准@1576]

Source structure [待校准@1577]

The source files for the site are all located under the source subdirectory. Templates for various sphinx plugins are located under source/_templates. The root directory contains configuration and files required to locally build the site for testing. [待校准@1578]

Building the site locally [待校准@1579]

Start by installing requirements located in the requirements.txt file: [待校准@1580]

The next command does a user-specific install, which requires ~/.local/bin/ to be added to $PATH: [待校准@1582]

pip3 install --user --upgrade -r requirements.txt

pip3 install --user --upgrade -r requirements.txt

python -m pip install --user --upgrade -r requirements.txt

In order for Sphinx to be able to generate diagrams, the dot command must be available. [待校准@1585]

sudo apt update ; sudo apt install graphviz

brew install graphviz

Download an installer from the Graphviz Download page and install it. Make sure to allow the installer to add it to the Windows %PATH%, otherwise Sphinx will not be able to find it. [待校准@1586]

Building the site for one branch [待校准@1587]

To build the site for just this branch, type make html at the top-level of the repository. This is the recommended way to test out local changes. [待校准@1588]

make html

The build process can take some time. To see the output, open build/html/index.html in your browser. [待校准@1589]

You can also run the documentation tests locally (using doc8) with the following command: [待校准@1590]

make test

Building the site for all branches [待校准@1591]

To build the site for all branches, type make multiversion from the rolling branch. This has two drawbacks: [待校准@1592]

  1. The multiversion plugin doesn't understand how to do incremental builds, so it always rebuilds everything. This can be slow. [待校准@1593]

  2. When typing make multiversion, it will always check out exactly the branches listed in the conf.py file. That means that local changes will not be shown. [待校准@1594]

To show local changes in the multiversion output, you must first commit the changes to a local branch. Then you must edit the conf.py file and change the smv_branch_whitelist variable to point to your branch. [待校准@1595]

Checking for broken links [待校准@1596]

To check for broken links on the site, run: [待校准@1597]

make linkcheck

This will check the entire site for broken links, and output the results to the screen and build/linkcheck. [待校准@1598]

Writing pages [待校准@1599]

The ROS 2 documentation website uses the reStructuredText format, which is the default plaintext markup language used by Sphinx. This section is a brief introduction to reStructuredText concepts, syntax, and best practices. [待校准@1600]

You can refer to reStructuredText User Documentation for a detailed technical specification. [待校准@1601]

Table of Contents [@1329]

There are two types of directives used for the generation of a table of contents, .. toctree:: and .. contents::. The .. toctree:: is used in top-level pages like Tutorials.rst to set ordering and visibility of its child pages. This directive creates both left navigation panel and in-page navigation links to the child pages listed. It helps readers to understand the structure of separate documentation sections and navigate between pages. [待校准@1602]

.. toctree::
   :maxdepth: 1

The .. contents:: directive is used for the generation of a table of contents for that particular page. It parses all present headings in a page and builds an in-page nested table of contents. It helps readers to see an overview of the content and navigate inside a page. [待校准@1603]

The .. contents:: directive supports the definition of maximum depth of nested sections. Using :depth: 2 will only show Sections and Subsections in the table of contents. [待校准@1604]

.. contents:: Table of Contents
   :depth: 2
   :local:

Headings [待校准@1605]

There are four main Heading types used in the documentation. Note that the number of symbols has to match the length of the title. [待校准@1606]

Page Title Header
=================

Section Header
--------------

2 Subsection Header
^^^^^^^^^^^^^^^^^^^

2.4 Subsubsection Header
~~~~~~~~~~~~~~~~~~~~~~~~

We usually use one digit for numbering subsections and two digits (dot separated) for numbering subsubsections in Tutorials and How-To-Guides. [待校准@1607]

Lists [待校准@1608]

Stars * are used for listing unordered items with bullet points and number sign #. is used for listing numbered items. Both of them support nested definitions and will render accordingly. [待校准@1609]

* bullet point

  * bullet point nested
  * bullet point nested

* bullet point

#. first listed item
#. second lited item

Code Formatting [待校准@1610]

In-text code can be formatted using backticks for showing highlighted code. [待校准@1611]

In-text code can be formatted using ``backticks`` for showing ``highlighted`` code.

Code blocks inside a page need to be captured using .. code-block:: directive. .. code-block:: supports code highlighting for syntaxes like C++, YAML, console, bash, and more. Code inside the directive needs to be indented. [待校准@1612]

.. code-block:: C++

   int main(int argc, char** argv)
   {
      rclcpp::init(argc, argv);
      rclcpp::spin(std::make_shared<ParametersClass>());
      rclcpp::shutdown();
      return 0;
   }

Images [待校准@1613]

Images can be inserted using the .. image:: directive. [待校准@1614]

.. image:: images/turtlesim_follow1.png

References and Links [待校准@1615]

External links [待校准@1616]

The syntax of creating links to external web pages is shown below. [待校准@1617]

`ROS Docs <https://docs.ros.org>`_

The above link will appear as ROS Docs. Note the underscore after the final single quote. [待校准@1618]

Internal links [待校准@1619]

The :doc: directive is used to create in-text links to other pages. [待校准@1620]

:doc:`Quality of Service <../Tutorials/Quality-of-Service>`

Note that the relative path to the file is used. [待校准@1621]

The ref directive is used to make links to specific parts of a page. These could be headings, images or code sections inside the current or different page. [待校准@1622]

Definition of explicit target right before the desired object is required. In the example below, the target is defined as _talker-listener one line before the heading Try some examples. [待校准@1623]

.. _talker-listener:

Try some examples
-----------------

Now the link from any page in the documentation to that header can be created. [待校准@1624]

:ref:`talker-listener demo <talker-listener>`

This link will navigate a reader to the target page with an HTML anchor link #talker-listener. [待校准@1625]

Macros can be used to simplify writing documentation that targets multiple distributions. [待校准@1627]

Use a macro by including the macro name in curly braces. For example, when generating the docs for Rolling on the rolling branch: [待校准@1628]

Use [待校准@1629]

Becomes (for Rolling) [待校准@1630]

Example [待校准@1631]

{DISTRO} [待校准@1632]

rolling [待校准@1633]

ros-{DISTRO}-pkg [待校准@1634]

{DISTRO_TITLE} [待校准@1635]

Rolling [待校准@1636]

ROS 2 {DISTRO_TITLE} [待校准@1637]

{DISTRO_TITLE_FULL} [待校准@1638]

Rolling Ridley [待校准@1639]

ROS 2 {DISTRO_TITLE_FULL} [待校准@1640]

{REPOS_FILE_BRANCH} [待校准@1641]

rolling [待校准@1633]

git checkout {REPOS_FILE_BRANCH} [待校准@1642]

The same file can be used on multiple branches (i.e., for multiple distros) and the generated content will be distro-specific. [待校准@1643]
