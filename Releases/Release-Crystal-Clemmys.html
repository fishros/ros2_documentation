Changes since the Bouncy release [待校准@14253]

Changes since the Bouncy Bolson release: [待校准@14254]

  • geometry2 - tf2_ros::Buffer API Change [待校准@14255]

    tf2_ros::Buffer now uses rclcpp::Time, with the constructor requiring a shared_ptr to a rclcpp::Clock instance. See https://github.com/ros2/geometry2/pull/67 for details, with example usage: [待校准@14256]

    #include <tf2_ros/transform_listener.h>
#include <rclcpp/rclcpp.hpp>
...
# Assuming you have a rclcpp::Node my_node
tf2_ros::Buffer buffer(my_node.get_clock());
tf2_ros::TransformListener tf_listener(buffer);

  • All rclcpp and rcutils logging macros require semicolons. [待校准@14257]

    See https://github.com/ros2/rcutils/issues/113 for details. [待校准@14258]

  • rcutils_get_error_string_safe() and rcl_get_error_string_safe() have been replaced with rcutils_get_error_string().str and rcl_get_error_string().str. [待校准@14259]

    See https://github.com/ros2/rcutils/pull/121 for details. [待校准@14260]

  • rmw - rmw_init API Change [待校准@14261]

    There are two new structs, the rcl_context_t and the rcl_init_options_t, which are used with rmw_init. The init options struct is used to pass options down to the middleware and is an input to rmw_init. The context is a handle which is an output of rmw_init function is used to identify which init-shutdown cycle each entity is associated with, where an "entity" is anything created like a node, guard condition, etc. [待校准@14262]

    This is listed here because maintainers of alternative rmw implementations will need to implement these new functions to have their rmw implementation work in Crystal. [待校准@14263]

    This is the function that had a signature change: [待校准@14264]

    • rmw_init [待校准@14265]

    Additionally, there are these new functions which need to be implemented by each rmw implementation: [待校准@14266]

    • rmw_shutdown [待校准@14267]

    • rmw_init_options_init [待校准@14268]

    • rmw_init_options_copy [待校准@14269]

    • rmw_init_options_fini [待校准@14270]

    Here's an example of what minimally needs to be changed in an rmw implementation to adhere to this API change: [待校准@14271]

    • rmw_fastrtps pr [待校准@14272]

  • rcl - rcl_init API Change [待校准@14273]

    Like the rmw change above, there's two new structs in rcl called rcl_context_t and rcl_init_options_t. The init options are passed into rcl_init as an input and the context is passed in as an output. The context is used to associate all other rcl entities to a specific init-shutdown cycle, effectively making init and shutdown no longer global functions, or rather those functions no longer use an global state and instead encapsulate all state within the context type. [待校准@14274]

    Any maintainers of a client library implementation (that also uses rcl under the hood) will need to make changes to work with Crystal. [待校准@14275]

    These functions were removed: [待校准@14276]

    • rcl_get_global_arguments [待校准@14277]

    • rcl_get_instance_id [待校准@14278]

    • rcl_ok [待校准@14279]

    These functions had signature changes: [待校准@14280]

    • rcl_init [待校准@14281]

    • rcl_shutdown [待校准@14282]

    • rcl_guard_condition_init [待校准@14283]

    • rcl_guard_condition_init_from_rmw [待校准@14284]

    • rcl_node_init [待校准@14285]

    • rcl_timer_init [待校准@14286]

    These are the new functions and types: [待校准@14287]

    • rcl_context_t [待校准@14288]

    • rcl_get_zero_initialized_context [待校准@14289]

    • rcl_context_fini [待校准@14290]

    • rcl_context_get_init_options [待校准@14291]

    • rcl_context_get_instance_id [待校准@14292]

    • rcl_context_is_valid [待校准@14293]

    • rcl_init_options_t [待校准@14294]

    • rcl_get_zero_initialized_init_options [待校准@14295]

    • rcl_init_options_init [待校准@14296]

    • rcl_init_options_copy [待校准@14297]

    • rcl_init_options_fini [待校准@14298]

    • rcl_init_options_get_rmw_init_options [待校准@14299]

    • rcl_node_is_valid_except_context [待校准@14300]

    • rcl_publisher_get_context [待校准@14301]

    • rcl_publisher_is_valid_except_context [待校准@14302]

    These new and changed functions will impact how you handle init and shutdown in your client library. For examples, look at the following rclcpp and rclpy PR's: [待校准@14303]

    • rclcpp [待校准@14304]

    • rclpy [待校准@14305]

    However, you may just continue to offer a single, global init and shutdown in your client library, and just store a single global context object. [待校准@14306]

Known Issues [待校准@14201]

  • A race condition in Fast-RTPS 1.7.0 may cause messages to drop under stress (Issue). [待校准@14307]

  • Using the TRANSIENT_LOCAL QoS setting with rmw_fastrtps_cpp can crash applications with large messages (Issue). [待校准@14308]

  • Cross-vendor communication between rmw_fastrtps_cpp and other implementations is not functioning on Windows (Issue). [待校准@14309]

  • When using OpenSplice (version < 6.9.190227) on macOS and Windows you might experience naming conflicts when when referencing field types with names from other packages if the same name also exist in the current package (Issue). By updating to a newer OpenSplice version as well as at least the third patch release of Crystal the problem should be resolved. On Linux updating to the latest Debian packages will include the newest OpenSplice version. [待校准@14310]
