与Bouncy版本发布相比的更改 [14253]

与 Bouncy Bolson 版本发布相比的更改： [14254]

  • geometry2 - tf2_ros::Buffer API 更改 [14255]

    tf2_ros::Buffer 现在使用 rclcpp::Time，构造函数需要一个指向 rclcpp::Clock 实例的 shared_ptr。有关详细信息，请参见 https://github.com/ros2/geometry2/pull/67，示例用法如下： [14256]

    #include <tf2_ros/transform_listener.h>
#include <rclcpp/rclcpp.hpp>
...
# Assuming you have a rclcpp::Node my_node
tf2_ros::Buffer buffer(my_node.get_clock());
tf2_ros::TransformListener tf_listener(buffer);

  • 所有 rclcpp 和 rcutils 的日志宏都需要分号。 [14257]

    有关详细信息，请参见 https://github.com/ros2/rcutils/issues/113。 [14258]

  • rcutils_get_error_string_safe() 和 rcl_get_error_string_safe() 已被替换为 rcutils_get_error_string().str 和 rcl_get_error_string().str。 [14259]

    有关详细信息，请参见 https://github.com/ros2/rcutils/pull/121。 [14260]

  • rmw - rmw_init API 变更 [14261]

    引入了两个新的结构体：rcl_context_t 和 rcl_init_options_t，它们与 rmw_init 一起使用。初始化选项结构体用于向中间件传递选项，并作为 rmw_init 的输入参数。上下文是一个句柄，是 rmw_init 函数的输出，用于标识每个实体与哪个初始化-关闭周期相关联，其中“实体”可以是节点、守护条件等。 [14262]

    将此列在此处是因为替代 rmw 实现的维护人员需要实现这些新函数，以使其 rmw 实现在 Crystal 中正常工作。 [14263]

    这是函数的签名发生变化的地方： [14264]

    • rmw_init [14265]

    此外，每个 rmw 实现都需要实现以下这些新函数： [14266]

    • rmw_shutdown [14267]

    • rmw_init_options_init [14268]

    • rmw_init_options_copy [14269]

    • rmw_init_options_fini [14270]

    以下是一个示例，最小限度地更改 rmw 实现以符合此 API 更改： [14271]

    • rmw_fastrtps pr [14272]

  • rcl - rcl_init API变更 [14273]

    和上面的``rmw``变更一样，在``rcl``中引入了两个新的结构体，分别是``rcl_context_t``和``rcl_init_options_t``。初始化选项被作为输入参数传递给``rcl_init``函数，而上下文则作为输出参数传递。上下文用于将所有其他的rcl实体关联到特定的初始化-关闭周期，有效地使初始化和关闭不再是全局函数，或者说这些函数不再使用全局状态，而是将所有状态封装在上下文类型中。 [14274]

    任何一个客户端库实现的维护者（也在底层使用``rcl``）都需要进行更改以适应Crystal。 [14275]

    以下函数已被移除: [14276]

    • rcl_get_global_arguments [14277]

    • rcl_get_instance_id [14278]

    • rcl_ok [14279]

    这些函数的签名发生了变化： [14280]

    • rcl_init [14281]

    • rcl_shutdown [14282]

    • rcl_guard_condition_init [14283]

    • rcl_guard_condition_init_from_rmw [14284]

    • rcl_node_init [14285]

    • rcl_timer_init [14286]

    以下是新的函数和类型: [14287]

    • rcl_context_t [14288]

    • rcl_get_zero_initialized_context [14289]

    • rcl_context_fini [14290]

    • rcl_context_get_init_options [14291]

    • rcl_context_get_instance_id [14292]

    • rcl_context_is_valid [14293]

    • rcl_init_options_t [14294]

    • rcl_get_zero_initialized_init_options [14295]

    • rcl_init_options_init [14296]

    • rcl_init_options_copy [14297]

    • rcl_init_options_fini [14298]

    • rcl_init_options_get_rmw_init_options [14299]

    • rcl_node_is_valid_except_context [14300]

    • rcl_publisher_get_context [14301]

    • rcl_publisher_is_valid_except_context [14302]

    这些新的和修改过的函数将影响您在客户端库中处理初始化和关闭的方式。有关示例，请查看以下 rclcpp 和 rclpy PR： [14303]

    • rclcpp [14304]

    • rclpy [14305]

    但是，您可以继续在您的客户端库中提供单个全局初始化和关闭，并只存储一个全局上下文对象。 [14306]

已知问题 [14201]

  • Fast-RTPS 1.7.0 中的竞争条件可能会在压力下导致消息丢失（Issue）。 [14307]

  • 在使用 rmw_fastrtps_cpp 的 TRANSIENT_LOCAL QoS 设置时，可能会导致具有大消息的应用程序崩溃（Issue）。 [14308]

  • 在Windows上，rmw_fastrtps_cpp`与其他实现之间的跨供应商通信不起作用（`问题）。 [14309]

  • 在macOS和Windows上使用OpenSplice（版本 < 6.9.190227）时，如果引用其他软件包中具有相同名称的字段类型，可能会遇到命名冲突，如果当前软件包中也存在相同的名称（问题）。通过升级到更新版本的OpenSplice以及至少Crystal的第三个补丁版本，问题应该得到解决。在Linux上，更新到最新的Debian软件包将包含最新的OpenSplice版本。 [14310]
