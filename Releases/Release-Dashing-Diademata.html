Dashing Diademata is primarily supported on the following platforms: [待校准@6789]

Tier 1 platforms: [待校准@6711]

  • Ubuntu 18.04 (Bionic): amd64 and arm64 [待校准@6790]

  • Mac macOS 10.12 (Sierra) [待校准@6643]

  • Windows 10 (Visual Studio 2019) [待校准@6791]

Tier 2 platforms: [待校准@6712]

  • Ubuntu 18.04 (Bionic): arm32 [待校准@6792]

Tier 3 platforms: [待校准@6793]

  • Debian Stretch (9): amd64, arm64 and arm32 [待校准@6794]

  • OpenEmbedded Thud (2.6) / webOS OSE: arm32 and x86 [待校准@6795]

New features in this ROS 2 release [待校准@6646]

A few features and improvements we would like to highlight: [待校准@6799]

  • Components are now the recommended way to write your node. They can be used standalone as well as being composed within a process and both ways are fully support from launch files. [待校准@6800]

  • The intra-process communication (C++ only) has been improved - both in terms of latency as well as minimizing copies. [待校准@6801]

  • The Python client library has been updated to match most of the C++ equivalent and some important bug fixes and improvements have landed related to memory usage and performance. [待校准@6802]

  • Parameters are now a complete alternative to dynamic_reconfigure from ROS 1 including constraints like ranges or being read-only. [待校准@6803]

  • By relying on (a subset of) IDL 4.2 for the message generation pipeline it is now possible to use .idl files (beside .msg / .srv / .action files). This change comes with support for optional UTF-8 encoding for ordinary strings as well as UTF-16 encoded multi-byte strings (see wide strings design article). [待校准@6804]

  • Command line tools related to actions and components. [待校准@6805]

  • Support for Deadline, Lifespan & Liveliness quality of service settings. [待校准@6806]

  • MoveIt 2 alpha release. [待校准@6807]

Please see the Dashing meta ticket on GitHub, which contains more information as well as references to specific tickets with additional details. [待校准@6808]

Changes since the Crystal release [待校准@6809]

Declaring Parameters [待校准@6810]

There have been some changes to the behavior of parameters starting in Dashing, which have also lead to some new API's and the deprecation of other API's. See the rclcpp and rclpy sections below for more information about API changes. [待校准@6811]

Getting and Setting Undeclared Parameters [待校准@6812]

As of Dashing, parameters now need to be declared before being accessed or set. [待校准@6813]

Before Dashing, you could call get_parameter(name) and get either a value, if it had been previously set, or a parameter of type PARAMETER_NOT_SET. You could also call set_parameter(name, value) at any point, even if the parameter was previously unset. [待校准@6814]

Since Dashing, you need to first declare a parameter before getting or setting it. If you try to get or set an undeclared parameter you will either get an exception thrown, e.g. ParameterNotDeclaredException, or in certain cases you will get an unsuccessful result communicated in a variety of ways (see specific functions for more details). [待校准@6815]

However, you can get the old behavior (mostly, see the note in the next paragraph) by using the allow_undeclared_parameters option when creating your node. You might want to do this in order to avoid code changes for now, or in order to fulfill some uncommon use cases. For example, a "global parameter server" or "parameter blackboard" may want to allow external nodes to set new parameters on itself without first declaring them, so it may use the allow_undeclared_parameters option to accomplish that. In most cases, however, this option is not recommended because it makes the rest of the parameter API less safe to bugs like parameter name typos and "use before set" logical errors. [待校准@6816]

Note that using allow_undeclared_parameters will get you most of the old behavior specifically for "get" and "set" methods, but it will not revert all the behavior changes related to parameters back to how it was for ROS Crystal. For that you need to also set the automatically_declare_parameters_from_overrides option to true, which is described below in Parameter Configuration using a YAML File. [待校准@6817]

Declaring a Parameter with a ParameterDescriptor [待校准@6818]

Another benefit to declaring your parameters before using them, is that it allows you to declare a parameter descriptor at the same time. [待校准@6819]

Now when declaring a parameter you may include a custom ParameterDescriptor as well as a name and default value. The ParameterDescriptor is defined as a message in rcl_interfaces/msg/ParameterDescriptor and contains meta data like description and constraints like read_only or integer_range. These constraints can be used to reject invalid values when setting parameters and/or as hints to external tools about what values are valid for a given parameter. The read_only constraint will prevent the parameter's value from changing after being declared, as well as prevent if from being undeclared. [待校准@6820]

For reference, here's a link to the ParameterDescriptor message as of the time of writing this: [待校准@6821]

https://github.com/ros2/rcl_interfaces/blob/0aba5a142878c2077d7a03977087e7d74d40ee68/rcl_interfaces/msg/ParameterDescriptor.msg#L1 [待校准@6822]

Parameter Configuration using a YAML File [待校准@6823]

As of Dashing, parameters in a YAML configuration file, e.g. passed to the node via the command line argument __params:=, are only used to override a parameter's default value when declaring the parameter. [待校准@6824]

Before Dashing, any parameters you passed via a YAML file would be implicitly set on the node. [待校准@6825]

Since Dashing, this is no longer the case, as parameters need to be declared in order to appear on the node to external observers, like ros2 param list. [待校准@6826]

The old behavior may be achieved using the automatically_declare_parameters_from_overrides option when creating a node. This option, if set to true, will automatically declare all parameters in the input YAML file when the node is constructed. This may be used to avoid major changes to your existing code or to serve specific use cases. For example, a "global parameter server" may want to be seeded with arbitrary parameters on launch, which it could not have declared ahead of time. Most of the time, however, this option is not recommended, as it may lead to setting a parameter in a YAML file with the assumption that the node will use it, even if the node does not actually use it. [待校准@6827]

In the future we hope to have a checker that will warn you if you pass a parameter to a node that it was not expecting. [待校准@6828]

The parameters in the YAML file will continue to influence the value of parameters when they are first declared. [待校准@6829]

ament_cmake [待校准@6830]

The CMake function ament_index_has_resource was returning either TRUE or FALSE. As of this release it returns either the prefix path in case the resource was found or FALSE. [待校准@6831]

If you are using the return value in a CMake condition like this: [待校准@6832]

ament_index_has_resource(var ...)
if(${var})

you need to update the condition to ensure it considers a string value as TRUE: [待校准@6833]

if(var)

rclcpp [待校准@6834]

Behavior Change for Node::get_node_names() [待校准@6835]

The function NodeGraph::get_node_names(), and therefore also Node::get_node_names(), now returns a std::vector<std::string> containing fully qualified node names with their namespaces included, instead of just the node names. [待校准@6836]

Changed the Way that Options are Passed to Nodes [待校准@6837]

Extended arguments (beyond name and namespace) to the rclcpp::Node() constructor have been replaced with a rclcpp::NodeOptions structure. See ros2/rclcpp#622 for details about the structure and default values of the options. [待校准@6838]

If you are using any of the extended arguments to rclcpp::Node() like this: [待校准@6839]

auto context = rclcpp::contexts::default_context::get_global_default_context();
std::vector<std::string> args;
std::vector<rclcpp::Parameter> params = { rclcpp::Parameter("use_sim_time", true) };
auto node = std::make_shared<rclcpp::Node>("foo_node", "bar_namespace", context, args, params);

You need to update to use the NodeOptions structure [待校准@6840]

std::vector<std::string> args;
std::vector<rclcpp::Parameter> params = { rclcpp::Parameter("use_sim_time", true) };
rclcpp::NodeOptions node_options;
node_options.arguments(args);
node_options.parameter_overrides(params);
auto node = std::make_shared<rclcpp::Node>("foo_node", "bar_namespace", node_options);

Changes to Creating Publishers and Subscriptions [待校准@6841]

There have been a few changes to creating publishers and subscriptions which are new in Dashing: [待校准@6842]

  • QoS settings are now passed using the new rclcpp::QoS class, and the API encourages the user to specify at least the history depth. [待校准@6843]

  • Options are now passed as an object, i.e. rclcpp::PublisherOptions and rclcpp::SubscriptionOptions. [待校准@6844]

All changes are backwards compatible (no code changes are required), but several existing call styles have been deprecated. Users are encouraged to update to the new signatures. [待校准@6845]

In the past, when creating a publisher or subscription, you could either not specify any QoS settings (e.g. just provide topic name for a publisher) or you could specify a "qos profile" data structure (of type rmw_qos_profile_t) with all the settings already set. Now you must use the new rclcpp::QoS object to specify your QoS and at least the history settings for your QoS. This encourages the user to specify a history depth when using KEEP_LAST, rather than defaulting it to a value that may or may not be appropriate. [待校准@6846]

In ROS 1, this was known as the queue_size and it was required in both C++ and Python. We're changing the ROS 2 API to bring this requirement back. [待校准@6847]

Also, any options which could previously be passed during creation of a publisher or subscription have now been encapsulated in an rclcpp::PublisherOptions and rclcpp::SubscriptionOptions class respectively. This allows for shorter signatures, more convenient use, and for adding new future options without breaking API. [待校准@6848]

Some signatures for creating publishers and subscribers are now deprecated, and new signatures have been added to allow you to use the new rclcpp::QoS and publisher/subscription option classes. [待校准@6849]

Changes to Creating Publishers, Subscriptions, and QoS Profiles [待校准@6882]

Prior to Dashing, you could optionally provide a QoSProfile object when creating a publisher or subscription. In an effort to encourage users to specify a history depth for message queues, we now require that a depth value or QoSProfile object is given when creating publishers or subscriptions. [待校准@6883]

To create a publisher, previously you would have written: [待校准@6884]

node.create_publisher(Empty, 'chatter')
# Or using a keyword argument for QoSProfile
node.create_publisher(Empty, 'chatter', qos_profile=qos_profile_sensor_data)

In Dashing, prefer the following API that provides a depth value or QoSProfile object as a third positional argument: [待校准@6885]

# Assume a history setting of KEEP_LAST with depth 10
node.create_publisher(Empty, 'chatter', 10)
# Or pass a QoSProfile object directly
node.create_publisher(Empty, 'chatter', qos_profile_sensor_data)

Likewise for subscriptions, previously you would have written: [待校准@6886]

node.create_subscription(BasicTypes, 'chatter', lambda msg: print(msg))
# Or using a keyword argument for QoSProfile
node.create_subscription(BasicTypes, 'chatter', lambda msg: print(msg), qos_profile=qos_profile_sensor_data)

In Dashing: [待校准@6887]

# Assume a history setting of KEEP_LAST with depth 10
node.create_subscription(BasicTypes, 'chatter', lambda msg: print(msg), 10)
# Or pass a QoSProfile object directly
node.create_subscription(BasicTypes, 'chatter', lambda msg: print(msg), qos_profile_sensor_data)

To ease the transition, users who do not use the new API will see deprecation warnings. [待校准@6888]

Furthermore, we also require that when constructing QoSProfile objects that a history policy and/or depth is set. If a history policy of KEEP_LAST is provided, then a depth argument is also required. For example, these calls are valid: [待校准@6889]

QoSProfile(history=QoSHistoryPolicy.RMW_QOS_POLICY_HISTORY_KEEP_ALL)
QoSProfile(history=QoSHistoryPolicy.RMW_QOS_POLICY_HISTORY_KEEP_LAST, depth=10)
QoSProfile(depth=10)  # equivalent to the previous line

And these calls will cause a deprecation warning: [待校准@6890]

QoSProfile()
QoSProfile(reliability=QoSReliabilityPolicy.RMW_QOS_POLICY_RELIABILITY_BEST_EFFORT)
# KEEP_LAST but no depth
QoSProfile(history=QoSHistoryPolicy.RMW_QOS_POLICY_HISTORY_KEEP_LAST)

See the issue and pull request related to introducing this change for more details: [待校准@6891]

rosidl [待校准@6904]

Until Crystal each message generator package registered itself using the ament_cmake extension point rosidl_generate_interfaces and was passed a set of .msg / .srv / .action files. As of Dashing the message generation pipeline is based on .idl files instead. [待校准@6905]

Any message generator package needs to change and register itself using the new extension point rosidl_generate_idl_interfaces which passes only .idl files instead. The message generators for the commonly supported languages C, C++, and Python as well as the typesupport packages for introspection, Fast RTPS, Connext and OpenSplice have already been updated (see ros2/rosidl#334). The CMake code calling rosidl_generate_interfaces() can either pass .idl files directly or pass .msg / .srv / .action which will then internally be converted into .idl files before being passed to each message generator. [待校准@6906]

The format of .msg / .srv / .action files is not being evolved in the future. The mapping between .msg / .srv / .action files and .idl files is described in this design article. A second design article describes the supported features in .idl files. In order to leverage any of the new features existing interfaces need to be converted (e.g. using the command line tools msg2idl / srv2idl / action2idl). [待校准@6907]

To distinguish same type names, but with different namespaces, the introspection structs now contain a namespace field that replaces the package name (see ros2/rosidl#335). [待校准@6908]

Mapping of char in .msg files [待校准@6909]

In ROS 1 char has been deprecated for a long time and is being mapped to uint8. In ROS 2 until Crystal char was mapped to a single character (char in C / C++, str with length 1 in Python) in an effort to provide a more natural mapping. As of Dashing the ROS 1 semantic has been restored and char maps to uint8 again. [待校准@6910]

rosidl_generator_cpp [待校准@6911]

The C++ data structures generated for messages, services and actions provide setter methods for each field. Until Crystal each setter returned a pointer to the data structure itself to enable the named parameter idiom. As of Dashing these setters return a reference instead since that seems to be the more common signature as well as it clarifies that the returned value can't be a nullptr. [待校准@6912]

rosidl_generator_py [待校准@6913]

Until Crystal an array (fixed size) or sequence (dynamic size, optionally with an upper boundary) field in a message was stored as a list in Python. As of Dashing the Python type for arrays / sequences of numeric values has been changed: [待校准@6914]

  • an array of numeric values is stored as a numpy.ndarray (the dtype is chosen to match the type of the numeric value) [待校准@6915]

  • a sequence of numeric values is stored as an array.array (the typename is chosen to match the type of the numeric value) [待校准@6916]

As before an array / sequence of non-numeric types is still represented as a list in Python. [待校准@6917]

This change brings a number of benefits: [待校准@6918]

  • The new data structures ensure that each item in the array / sequence complies with the value range restrictions of the numeric type. [待校准@6919]

  • The numeric values can be stored more efficiently in memory which avoid the overhead of Python objects for each item. [待校准@6920]

  • The memory layout of both data structures allows to read and write all items of the array / sequence in a single operation which makes the conversion from and to Python significantly faster / more efficient. [待校准@6921]
