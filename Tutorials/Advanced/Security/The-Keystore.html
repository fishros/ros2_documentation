Understanding the security keystore [待校准@3571]

Background [待校准@3072]

The sros2 package can be used to create keys, certificates and policies necessary to enable ROS 2 security. However, the security configuration is extrememly flexible. A basic understanding of the ROS 2 Security Keystore will allow integration with an existing PKI (Public Key Infrastructure) and managment of sensitive key materials consistent with organizational policies. [待校准@3573]

Security Artifact Locations [待校准@3574]

With communications security enabled in the prior tutorial, let's take a look at the files which were created when security was enabled. These are the files which make encryption possible. [待校准@3575]

The sros2 utilities (ros2 security ...) separate files into public, private and enclave key materials. [待校准@3576]

ROS uses the directory defined by the environmental variable ROS_SECURITY_KEYSTORE as the keystore. For this tutorial, we use the directory ~/sros2_demo/demo_keystore. [待校准@3577]

Public Key Materials [待校准@3578]

You will find three encryption certificates in the public directory at ~/sros2_demo/demo_keys/public; however, the identity and permissions certificates are actually just a link to the Certificate Authority (CA) certificate. [待校准@3579]

In a public key infrastructure, the Certificate Authority acts as a trust anchor: it validates the identities and permissions of participants. For ROS, that means all the nodes that participate in the ROS graph (which may extend to an entire fleet of individual robots). By placing the Certificate Authority's certificate (ca.cert.pem) in the proper location on the robot, all ROS nodes can establish mutual trust with other nodes using the same Certificate Authority. [待校准@3580]

Although in our tutorials we create a Certificate Authority on-the-fly, in a production system this should be done according to a pre-defined security plan. Typically the Certificate Authority for a production system will be created off-line, and placed on the robot during initial setup. It may be unique for each robot, or shared across a fleet of robots all intended to trust each other. [待校准@3581]

DDS (and ROS, by extension) supports separation of identity and permission trust chains, so each function has its own certificate authority. In most cases a ROS system security plan does not require a separation between these duties, so the security utilities generate a single Certificate Authority which is used for both identity and permissions. [待校准@3582]

Use openssl to view this x509 certificate and display it as text: [待校准@3583]

openssl x509 -in ca.cert.pem -text -noout

Some things to note about this CA certificate: [待校准@3585]

  • The certificate subject name sros2testCA is the default provided by the sros2 utilities. [待校准@3586]

  • As a Root Certificate Authority, this is a self-signed certificate; i.e., it is signed using its own private key. [待校准@3589]

Since this is a public certificate, it can be freely copied as needed to establish trust throughout your ROS system. [待校准@3590]

Private Key Materials [待校准@3591]

Private key materials can be found in the keystore directory ~/sros2_demo/demo_keys/private. Similar to the public directory, this contains one certificate authority key ca.key.pem and symbolic links to it to be used as both an Identity and a Permissions CA private key. [待校准@3592]

警告

Protect this private key and create a secure backup of it! [待校准@3593]

This is the private key associated with the public Certificate Authority which serves as the anchor for all security in your ROS system. You will use it to modify encryption policies for the ROS graph and to add new ROS participants. Depending upon your robot's security needs, the key can be protected with access permissions and locked down to another account, or it can be moved off the robot entirely and onto another system or device. If the file is lost, you will be unable to change access permissions and add new participants to the system. Similarly, any user or process with access to the file has the ability to modify system policies and participants. [待校准@3594]

This file is only required for configuring the robot, but is not needed for the robot to run. It can safely be stored offline in another system or removable media. [待校准@3595]

The sros2 utilities use elliptic curve cryptograpy rather than RSA for improved security and reduced key size. Use the following command to show details about this elliptic curve private key: [待校准@3596]

openssl ec -in ca.key.pem -text -noout

Domain Governance Policy [待校准@3599]

Find the domain governance policy in the enclave directory within the keystore, ~/sros2_demo/demo_keys/enclaves. The enclave directory contains XML governance policy document governance.xml, as well as a copy of the document which has been signed by the Permissions CA as governance.p7s. [待校准@3600]

The governance.p7s file contains domain-wide settings such as how to handle unauthenticated participants, whether to encrypt discovery, and default rules for access to topics. [待校准@3601]

Use the following command to validate the S/MIME signature of the governance file: [待校准@3602]

openssl smime -verify -in governance.p7s -CAfile ../public/permissions_ca.cert.pem

This command will print out the XML document, and the last line will be Verification successful to show that the document was properly signed by the Permissions CA. [待校准@3603]

Security Enclaves [待校准@3604]

Secure processes (typically ROS nodes) run within a security enclave. In the simplest case, all the processes can be consolidated into the same enclave, and all processes will then use the same security policy. However, to apply different policies to different processes, the processes can use different security enclaves when starting. For more details about security enclaves, see the design document. The security enclave is specifed by using the ROS argument --enclave when running a node. [待校准@3605]

Each security enclave requires six files in order to enable security. Each file must be named as defined below, and as outlined in the DDS Security standard. In order to avoid having mulitple copies of the same files, the sros2 utilities create links for each enclave to the single governance policy, the Identity CA and Permissions CA descibed above. [待校准@3606]

See the following six files within the listener enclave. Three are specific to this enclave, while three are generic to this ROS system: [待校准@3607]

  • key.pem, the private key used to encrypt and decrypt within this enclave [待校准@3608]

  • cert.pem, the public certificate for this enclave; this certificate has been signed by the Identity CA [待校准@3609]

  • permissions.p7s, the permissions for this enclave; this file has been signed with the Permissions CA [待校准@3610]

  • governance.p7s, a link to the signed security policy file for this domain [待校准@3611]

  • identity_ca.cert.pem, a link to the Identity CA for this domain [待校准@3612]

The private encryption key key.pem should be protected according to your security plan. This key encrypts, decrypts and validates communications within this specific enclave. Should the key be lost or stolen, revoke the key and create a new identity for this enclave. [待校准@3614]

The file permissions.xml has also been created in this directory and can be used to recreate the signed permissions file. However, this file is not required to enable security since DDS uses the signed version of the file instead. [待校准@3615]

What is the difference between launching the listener and launching the talker? [待校准@3622]

The listener fails to launch and throws an error. When the permissions.p7s file was modified--however minor--the file's signature became invalid. A node will not launch with security enabled and enforced when the permissions file is invalid. [待校准@3623]

The talker will start as expected. It uses the permissions.p7s file in a different enclave, and the file is still valid. [待校准@3624]

What command lets you check to see if the signature on the modified permissions.p7s file is valid? [待校准@3625]

Check that permissions.p7s has been properly signed by the Permissions CA using the openssl smime command: [待校准@3626]

openssl smime -verify -in permissions.p7s -CAfile permissions_ca.cert.pem
