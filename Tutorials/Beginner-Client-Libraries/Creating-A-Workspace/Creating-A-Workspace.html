Creating a workspace [待校准@4428]

Goal: Create a workspace and learn how to set up an overlay for development and testing. [待校准@4429]

Tutorial level: Beginner [待校准@3951]

Time: 20 minutes [待校准@3069]

Background [待校准@3072]

A workspace is a directory containing ROS 2 packages. Before using ROS 2, it's necessary to source your ROS 2 installation workspace in the terminal you plan to work in. This makes ROS 2's packages available for you to use in that terminal. [待校准@4430]

You also have the option of sourcing an "overlay" - a secondary workspace where you can add new packages without interfering with the existing ROS 2 workspace that you're extending, or "underlay". Your underlay must contain the dependencies of all the packages in your overlay. Packages in your overlay will override packages in the underlay. It's also possible to have several layers of underlays and overlays, with each successive overlay using the packages of its parent underlays. [待校准@4431]

Prerequisites [待校准@3124]

  • ROS 2 installation [待校准@4432]

  • colcon installation [待校准@4433]

  • git installation [待校准@4434]

  • turtlesim installation [待校准@4435]

  • Have rosdep installed [待校准@4436]

  • Understanding of basic terminal commands (here's a guide for Linux) [待校准@4437]

  • Text editor of your choice [待校准@4438]

Tasks [待校准@3282]

1 Source ROS 2 environment [待校准@4439]

Your main ROS 2 installation will be your underlay for this tutorial. (Keep in mind that an underlay does not necessarily have to be the main ROS 2 installation.) [待校准@4440]

Depending on how you installed ROS 2 (from source or binaries), and which platform you're on, your exact source command will vary: [待校准@4441]

source /opt/ros/humble/setup.bash

. ~/ros2_install/ros2-osx/setup.bash

Remember to use a x64 Native Tools Command Prompt for VS 2019 for executing the following commands, as we are going to build a workspace. [待校准@4442]

call C:\dev\ros2\local_setup.bat

Consult the installation guide you followed if these commands don't work for you. [待校准@4443]

2 Create a new directory [待校准@4444]

Best practice is to create a new directory for every new workspace. The name doesn't matter, but it is helpful to have it indicate the purpose of the workspace. Let's choose the directory name ros2_ws, for "development workspace": [待校准@4445]

mkdir -p ~/ros2_ws/src
cd ~/ros2_ws/src

mkdir -p ~/ros2_ws/src
cd ~/ros2_ws/src

md \ros2_ws\src
cd \ros2_ws\src

Another best practice is to put any packages in your workspace into the src directory. The above code creates a src directory inside ros2_ws and then navigates into it. [待校准@4446]

3 Clone a sample repo [待校准@4447]

Ensure you're still in the ros2_ws/src directory before you clone. [待校准@4448]

In the rest of the beginner developer tutorials, you will create your own packages, but for now you will practice putting a workspace together using existing packages. [待校准@4449]

If you went through the Beginner: CLI Tools tutorials, you'll be familiar with turtlesim, one of the packages in ros_tutorials. [待校准@4450]

A repo can have multiple branches. You need to check out the one that targets your installed ROS 2 distro. When you clone this repo, add the -b argument followed by that branch. [待校准@4451]

In the ros2_ws/src directory, run the following command: [待校准@4452]

git clone https://github.com/ros/ros_tutorials.git -b humble

Now ros_tutorials is cloned in your workspace. The ros_tutorials repository contains the turtlesim package, which we'll use in the rest of this tutorial. The other packages in this repository are not built because they contain a COLCON_IGNORE file. [待校准@4453]

So far you have populated your workspace with a sample package, but it isn't a fully-functional workspace yet. You need to resolve the dependencies first and then build the workspace. [待校准@4454]

4 Resolve dependencies [待校准@4455]

Before building the workspace, you need to resolve the package dependencies. You may have all the dependencies already, but best practice is to check for dependencies every time you clone. You wouldn't want a build to fail after a long wait only to realize that you have missing dependencies. [待校准@4456]

From the root of your workspace (ros2_ws), run the following command: [待校准@4457]

# cd if you're still in the ``src`` directory with the ``ros_tutorials`` clone
cd ..
rosdep install -i --from-path src --rosdistro humble -y

rosdep only runs on Linux, so you can skip ahead to section "5 Build the workspace with colcon". [待校准@4458]

rosdep only runs on Linux, so you can skip ahead to section "5 Build the workspace with colcon". [待校准@4458]

If you installed ROS 2 on Linux from source or the "fat" archive, you will need to use the rosdep command from their installation instructions. Here are the from-source rosdep section and the "fat" archive rosdep section. [待校准@4459]

If you already have all your dependencies, the console will return: [待校准@4460]

#All required rosdeps installed successfully

Packages declare their dependencies in the package.xml file (you will learn more about packages in the next tutorial). This command walks through those declarations and installs the ones that are missing. You can learn more about rosdep in another tutorial (coming soon). [待校准@4461]

5 Build the workspace with colcon [待校准@4462]

From the root of your workspace (ros2_ws), you can now build your packages using the command: [待校准@4463]

colcon build

colcon build

colcon build --merge-install

Windows doesn't allow long paths, so merge-install will combine all the paths into the install directory. [待校准@4397]

The console will return the following message: [待校准@4464]

Starting >>> turtlesim
Finished <<< turtlesim [5.49s]

Summary: 1 package finished [5.58s]

注解

Other useful arguments for colcon build: [待校准@4465]

  • --packages-up-to builds the package you want, plus all its dependencies, but not the whole workspace (saves time) [待校准@4466]

  • --symlink-install saves you from having to rebuild every time you tweak python scripts [待校准@4467]

  • --event-handlers console_direct+ shows console output while building (can otherwise be found in the log directory) [待校准@4468]

Once the build is finished, enter ls in the workspace root (~/ros2_ws) and you will see that colcon has created new directories: [待校准@4469]

build  install  log  src

The install directory is where your workspace's setup files are, which you can use to source your overlay. [待校准@4470]

6 Source the overlay [待校准@4471]

Before sourcing the overlay, it is very important that you open a new terminal, separate from the one where you built the workspace. Sourcing an overlay in the same terminal where you built, or likewise building where an overlay is sourced, may create complex issues. [待校准@4472]

In the new terminal, source your main ROS 2 environment as the "underlay", so you can build the overlay "on top of" it: [待校准@4473]

source /opt/ros/humble/setup.bash

. ~/ros2_install/ros2-osx/setup.bash

In this case you can use a normal command prompt, as we are not going to build any workspace in this terminal. [待校准@4474]

call C:\dev\ros2\local_setup.bat

Go into the root of your workspace: [待校准@4475]

cd ~/ros2_ws

cd ~/ros2_ws

cd \ros2_ws

In the root, source your overlay: [待校准@4476]

source install/local_setup.bash

. install/local_setup.bash

call install\setup.bat

注解

Sourcing the local_setup of the overlay will only add the packages available in the overlay to your environment. setup sources the overlay as well as the underlay it was created in, allowing you to utilize both workspaces. [待校准@4477]

So, sourcing your main ROS 2 installation's setup and then the ros2_ws overlay's local_setup, like you just did, is the same as just sourcing ros2_ws's setup, because that includes the environment of its underlay. [待校准@4478]

Now you can run the turtlesim package from the overlay: [待校准@4479]

ros2 run turtlesim turtlesim_node

But how can you tell that this is the overlay turtlesim running, and not your main installation's turtlesim? [待校准@4480]

Let's modify turtlesim in the overlay so you can see the effects: [待校准@4481]

  • You can modify and rebuild packages in the overlay separately from the underlay. [待校准@4482]

  • The overlay takes precedence over the underlay. [待校准@4483]

7 Modify the overlay [待校准@4484]

You can modify turtlesim in your overlay by editing the title bar on the turtlesim window. To do this, locate the turtle_frame.cpp file in ~/ros2_ws/src/ros_tutorials/turtlesim/src. Open turtle_frame.cpp with your preferred text editor. [待校准@4485]

On line 52 you will see the function setWindowTitle("TurtleSim");. Change the value "TurtleSim" to "MyTurtleSim", and save the file. [待校准@4486]

Return to the first terminal where you ran colcon build earlier and run it again. [待校准@4487]

Return to the second terminal (where the overlay is sourced) and run turtlesim again: [待校准@4488]

ros2 run turtlesim turtlesim_node

You will see the title bar on the turtlesim window now says "MyTurtleSim". [待校准@4489]

Even though your main ROS 2 environment was sourced in this terminal earlier, the overlay of your ros2_ws environment takes precedence over the contents of the underlay. [待校准@4490]

To see that your underlay is still intact, open a brand new terminal and source only your ROS 2 installation. Run turtlesim again: [待校准@4491]

ros2 run turtlesim turtlesim_node

You can see that modifications in the overlay did not actually affect anything in the underlay. [待校准@4492]

Summary [待校准@3353]

In this tutorial, you sourced your main ROS 2 distro install as your underlay, and created an overlay by cloning and building packages in a new workspace. The overlay gets prepended to the path, and takes precedence over the underlay, as you saw with your modified turtlesim. [待校准@4493]

Using overlays is recommended for working on a small number of packages, so you don't have to put everything in the same workspace and rebuild a huge workspace on every iteration. [待校准@4494]
