Overview [待校准@11174]

The logging subsystem in ROS 2 aims to deliver logging messages to a variety of targets, including: [待校准@11360]

  • To the console (if one is attached) [待校准@11361]

  • To log files on disk (if local storage is available) [待校准@11362]

  • To the /rosout topic on the ROS 2 network [待校准@11363]

By default, log messages in ROS 2 nodes will go out to the console (on stderr), to log files on disk, and to the /rosout topic on the ROS 2 network. All of the targets can be individually enabled or disabled on a per-node basis. [待校准@11364]

The rest of this document will go over some of the ideas behind the logging subsystem. [待校准@11365]

Severity level [待校准@11366]

Log messages have a severity level associated with them: DEBUG, INFO, WARN, ERROR or FATAL, in ascending order. [待校准@11367]

A logger will only process log messages with severity at or higher than a specified level chosen for the logger. [待校准@11368]

Each node has a logger associated with it that automatically includes the node's name and namespace. If the node's name is externally remapped to something other than what is defined in the source code, it will be reflected in the logger name. Non-node loggers can also be created that use a specific name. [待校准@11369]

Logger names represent a hierarchy. If the level of a logger named "abc.def" is unset, it will defer to the level of its parent named "abc", and if that level is also unset, the default logger level will be used. When the level of logger "abc" is changed, all of its descendants (e.g. "abc.def", "abc.ghi.jkl") will have their level impacted unless their level has been explicitly set. [待校准@11370]

APIs [待校准@11371]

These are the APIs that end users of the ROS 2 logging infrastructure should use, split up by client library. [待校准@11372]

  • RCLCPP_{DEBUG,INFO,WARN,ERROR,FATAL} - output the given printf-style message every time this line is hit [待校准@11373]

  • RCLCPP_{DEBUG,INFO,WARN,ERROR,FATAL}_ONCE - output the given printf-style message only the first time this line is hit [待校准@11374]

  • RCLCPP_{DEBUG,INFO,WARN,ERROR,FATAL}_EXPRESSION - output the given printf-style message only if the given expression is true [待校准@11375]

  • RCLCPP_{DEBUG,INFO,WARN,ERROR,FATAL}_FUNCTION - output the given printf-style message only if the given function returns true [待校准@11376]

  • RCLCPP_{DEBUG,INFO,WARN,ERROR,FATAL}_SKIPFIRST - output the given printf-style message all but the first time this line is hit [待校准@11377]

  • RCLCPP_{DEBUG,INFO,WARN,ERROR,FATAL}_THROTTLE - output the given printf-style message no more than the given rate in integer milliseconds [待校准@11378]

  • RCLCPP_{DEBUG,INFO,WARN,ERROR,FATAL}_SKIPFIRST_THROTTLE - output the given printf-style message no more than the given rate in integer milliseconds, but skip the first [待校准@11379]

  • RCLCPP_{DEBUG,INFO,WARN,ERROR,FATAL}_STREAM - output the given C++ stream-style message every time this line is hit [待校准@11380]

  • RCLCPP_{DEBUG,INFO,WARN,ERROR,FATAL}_STREAM_ONCE - output the given C++ stream-style message only the first time this line is hit [待校准@11381]

  • RCLCPP_{DEBUG,INFO,WARN,ERROR,FATAL}_STREAM_EXPRESSION - output the given C++ stream-style message only if the given expression is true [待校准@11382]

  • RCLCPP_{DEBUG,INFO,WARN,ERROR,FATAL}_STREAM_FUNCTION - output the given C++ stream-style message only if the given function returns true [待校准@11383]

  • RCLCPP_{DEBUG,INFO,WARN,ERROR,FATAL}_STREAM_SKIPFIRST - output the given C++ stream-style message all but the first time this line is hit [待校准@11384]

  • RCLCPP_{DEBUG,INFO,WARN,ERROR,FATAL}_STREAM_THROTTLE - output the given C++ stream-style message no more than the given rate in integer milliseconds [待校准@11385]

  • RCLCPP_{DEBUG,INFO,WARN,ERROR,FATAL}_STREAM_SKIPFIRST_THROTTLE - output the given C++ stream-style message no more than the given rate in integer milliseconds, but skip the first [待校准@11386]

Each of the above APIs takes an rclcpp::Logger object as the first argument. This can be pulled from the node API by calling node->get_logger() (recommended), or by constructing a stand-alone rclcpp::Logger object. [待校准@11387]

  • rcutils_logging_set_logger_level - Set the logging level for a particular logger name to the given severity level [待校准@11388]

  • rcutils_logging_get_logger_effective_level - Given a logger name, return the logger level (which may be unset) [待校准@11389]

  • logger.{debug,info,warning,error,fatal} - output the given Python string to the logging infrastructure. The calls accept the following keyword args to control behavior: [待校准@11390]

    • throttle_duration_sec - if not None, the duration of the throttle interval in floating-point seconds [待校准@11391]

    • skip_first - if True, output the message all but the first time this line is hit [待校准@11392]

    • once - if True, only output the message the first time this line is hit [待校准@11393]

  • rclpy.logging.set_logger_level - Set the logging level for a particular logger name to the given severity level [待校准@11394]

  • rclpy.logging.get_logger_effective_level - Given a logger name, return the logger level (which may be unset) [待校准@11395]

Configuration [待校准@11396]

Since rclcpp and rclpy use the same underlying logging infrastructure, the configuration options are the same. [待校准@11397]

Environment variables [待校准@11398]

The following environment variables control some aspects of the ROS 2 loggers. For each of the environment settings, note that this is a process-wide setting, so it applies to all nodes in that process. [待校准@11399]

  • ROS_LOG_DIR - Control the logging directory that is used for writing logging messages to disk (if that is enabled). If non-empty, use the exact directory as specified in this variable. If empty, use the contents of the ROS_HOME environment variable to construct a path of the form $ROS_HOME/.log. In all cases, the ~ character is expanded to the user's HOME directory. [待校准@11400]

  • ROS_HOME - Control the home directory that is used for various ROS files, including logging and config files. In the context of logging, this variable is used to construct a path to a directory for log files. If non-empty, use the contents of this variable for the ROS_HOME path. In all cases, the ~ character is expanded to the users's HOME directory. [待校准@11401]

  • RCUTILS_LOGGING_USE_STDOUT - Control what stream output messages go to. If this is unset or 0, use stderr. If this is 1, use stdout. [待校准@11402]

  • RCUTILS_LOGGING_BUFFERED_STREAM - Control whether the logging stream (as configured in RCUTILS_LOGGING_USE_STDOUT) should be line buffered or unbuffered. If this is unset, use the default of the stream (generally line buffered for stdout, and unbuffered for stderr). If this is 0, force the stream to be unbuffered. If this is 1, force the stream to be line buffered. [待校准@11403]

  • RCUTILS_COLORIZED_OUTPUT - Control whether colors are used when outputting messages. If unset, automatically determine based on the platform and whether the console is a TTY. If 0, force disable using colors for output. If 1, force enable using colors for output. [待校准@11404]

  • RCUTILS_CONSOLE_OUTPUT_FORMAT - Control the fields that are output for each log message. The available fields are: [待校准@11405]

    • {severity} - The severity level. [待校准@11406]

    • {name} - The name of the logger (may be empty). [待校准@11407]

    • {message} - The log message (may be empty). [待校准@11408]

    • {function_name} - The function name this was called from (may be empty). [待校准@11409]

    • {file_name} - The file name this was called from (may be empty). [待校准@11410]

    • {time} - The time in seconds since the epoch. [待校准@11411]

    • {time_as_nanoseconds} - The time in nanoseconds since the epoch. [待校准@11412]

    • {line_number} - The line number this was called from (may be empty). [待校准@11413]

    If no format is given, a default of [{severity}] [{time}] [{name}]: {message} is used. [待校准@11414]

Node creation [待校准@11415]

When initializing a ROS 2 node, it is possible to control some aspects of the behavior via node options. Since these are per-node options, they can be set differently for different nodes even when the nodes are composed into a single process. [待校准@11416]

  • log_levels - The log level to use for a component within this particular node. This can be set with the following: ros2 run demo_nodes_cpp talker --ros-args --log-level talker:=DEBUG [待校准@11417]

  • external_log_config_file - The external file to use to configure the backend logger. If it is NULL, the default configuration will be used. Note that the format of this file is backend-specific (and is currently unimplemented for the default backend logger of spdlog). This can be set with the following: ros2 run demo_nodes_cpp talker --ros-args --log-config-file log-config.txt [待校准@11418]

  • log_stdout_disabled - Whether to disable writing log messages to the console. This can be done with the following: ros2 run demo_nodes_cpp talker --ros-args --disable-stdout-logs [待校准@11419]

  • log_rosout_disabled - Whether to disable writing log messages out to /rosout. This can significantly save on network bandwidth, but external observers will not be able to monitor logging. This can be done with the following: ros2 run demo_nodes_cpp talker --ros-args --disable-rosout-logs [待校准@11420]

  • log_ext_lib_disabled - Whether to completely disable the use of an external logger. This may be faster in some cases, but means that logs will not be written to disk. This can be done with the following: ros2 run demo_nodes_cpp talker --ros-args --disable-external-lib-logs [待校准@11421]

Logging subsystem design [待校准@11422]

The image below shows the five main pieces to the logging subsystem and how they interact. [待校准@11423]

rcutils [待校准@11425]

rcutils has a logging implementation that can format log messages according to a certain format (see Configuration above), and output those log messages to a console. rcutils implements a complete logging solution, but allows higher-level components to insert themselves into the logging infrastructure in a dependency-injection model. This will become more evident when we talk about the rcl layer below. [待校准@11426]

Note that this is a per-process logging implementation, so anything that is configured at this level will affect the entire process, not just individual nodes. [待校准@11427]

rcl_logging_spdlog [待校准@11428]

rcl_logging_spdlog implements the rcl_logging_interface API, and thus provides external logging services to the rcl layer. In particular, the rcl_logging_spdlog implementation takes formatted log messages and writes them out to log files on disk using the spdlog library, typically within ~/.ros/log (though this is configurable; see Configuration above). [待校准@11429]

rcl [待校准@11430]

The logging subsystem in rcl uses rcutils and rcl_logging_spdlog to provide the bulk of the ROS 2 logging services. When log messages come in, rcl decides where to send them. There are 3 main places that logging messages can be delivered; an individual node may have any combination of them enabled: [待校准@11431]

  • To the console via the rcutils layer [待校准@11432]

  • To disk via the rcl_logging_spdlog layer [待校准@11433]

  • To the /rosout topic on the ROS 2 network via the RMW layer [待校准@11434]

rclcpp [待校准@11435]

This is the main ROS 2 C++ API which sits atop the rcl API. In the context of logging, rclcpp provides the RCLCPP_ logging macros; see APIs above for a complete list. When one of the RCLCPP_ macros runs, it checks the current severity level of the node against the severity level of the macro. If the severity level of the macro is greater than or equal to the node severity level, the message will be formatted and output to all of the places that are currently configured. Note that rclcpp uses a global mutex for log calls, so all logging calls within the same process end up being single-threaded. [待校准@11436]

rclpy [待校准@11437]

This is the main ROS 2 Python API which sits atop the rcl API. In the context of logging, rclpy provides the logger.debug-style functions; see APIs above for a complete list. When one of the logger.debug functions runs, it checks the current severity level of the node against the severity level of the macro. If the severity level of the macro is greater than or equal to the node severity level, the message will be formatted and output to all of the places that are currently configured. [待校准@11438]
