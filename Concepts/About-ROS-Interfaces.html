1. Background [待校准@11637]

ROS applications typically communicate through interfaces of one of three types: messages, services and actions. ROS 2 uses a simplified description language, the interface definition language (IDL), to describe these interfaces. This description makes it easy for ROS tools to automatically generate source code for the interface type in several target languages. [待校准@11638]

In this document we will describe the supported types: [待校准@11639]

  • msg: .msg files are simple text files that describe the fields of a ROS message. They are used to generate source code for messages in different languages. [待校准@11640]

  • srv: .srv files describe a service. They are composed of two parts: a request and a response. The request and response are message declarations. [待校准@11641]

  • action: .action files describe actions. They are composed of three parts: a goal, a result, and feedback. Each part is a message declaration itself. [待校准@11642]

2. Message description specification [待校准@11643]

Messages are described and defined in .msg files in the msg/ directory of a ROS package. .msg files are composed of two parts: fields and constants. [待校准@11644]

2.1 Fields [待校准@11645]

Each field consists of a type and a name, separated by a space, i.e: [待校准@11646]

fieldtype1 fieldname1
fieldtype2 fieldname2
fieldtype3 fieldname3

For example: [待校准@11647]

int32 my_int
string my_string

  • a built-in-type [待校准@11650]

2.1.2 Field names [待校准@11713]

Field names must be lowercase alphanumeric characters with underscores for separating words. They must start with an alphabetic character, and they must not end with an underscore or have two consecutive underscores. [待校准@11714]

2.1.3 Field default value [待校准@11715]

Default values can be set to any field in the message type. Currently default values are not supported for string arrays and complex types (i.e. types not present in the built-in-types table above; that applies to all nested messages). [待校准@11716]

Defining a default value is done by adding a third element to the field definition line, i.e: [待校准@11717]

fieldtype fieldname fielddefaultvalue

For example: [待校准@11647]

uint8 x 42
int16 y -2000
string full_name "John Doe"
int32[] samples [-200, -100, 0, 100, 200]

Note: [待校准@11718]

  • string values must be defined in single ' or double " quotes [待校准@11719]

  • currently string values are not escaped [待校准@11720]

2.2 Constants [待校准@11721]

Each constant definition is like a field description with a default value, except that this value can never be changed programatically. This value assignment is indicated by use of an equal '=' sign, e.g. [待校准@11722]

constanttype CONSTANTNAME=constantvalue

For example: [待校准@11647]

int32 X=123
int32 Y=-123
string FOO="foo"
string EXAMPLE='bar'

注解

Constants names have to be UPPERCASE [待校准@11723]

3. Service description specification [待校准@11724]

Services are described and defined in .srv files in the srv/ directory of a ROS package. [待校准@11725]

A service description file consists of a request and a response msg type, separated by ---. Any two .msg files concatenated with a --- are a legal service description. [待校准@11726]

Here is a very simple example of a service that takes in a string and returns a string: [待校准@11727]

string str
---
string str

We can of course get much more complicated (if you want to refer to a message from the same package you must not mention the package name): [待校准@11728]

#request constants
int8 FOO=1
int8 BAR=2
#request fields
int8 foobar
another_pkg/AnotherMessage msg
---
#response constants
uint32 SECRET=123456
#response fields
another_pkg/YetAnotherMessage val
CustomMessageDefinedInThisPackage value
uint32 an_integer

You cannot embed another service inside of a service. [待校准@11729]

4. New features in ROS 2 interfaces [待校准@11730]

The ROS 2 IDL is closely related to the ROS 1 IDL. Most existing ROS 1 .msg and .srv files should be usable as-is with ROS 2. Atop ROS 1's existing feature set, the ROS 2 IDL introduces some new features, namely: [待校准@11731]

  • bounded arrays: Whereas the ROS 1 IDL allows unbounded arrays (e.g., int32[] foo) and fixed-size arrays (e.g., int32[5] bar), the ROS 2 IDL further allows bounded arrays (e.g., int32[<=5] bat). There are use cases in which it's important to be able to place an upper bound on the size of an array without committing to always using that much space (e.g., in a real-time system in which you need to preallocate all memory that will be used during execution). [待校准@11732]

  • bounded strings: Whereas the ROS 1 IDL allows unbounded strings (e.g., string foo), the ROS 2 IDL further allows bounded strings (e.g., string<=5 bar). [待校准@11733]

  • default values: Whereas the ROS 1 IDL allows constant fields (e.g., int32 X=123), the ROS 2 IDL further allows default values to be specified (e.g., int32 X 123). The default value is used when constructing a message/service object and can be subsequently overridden by assigning to the field. You can also specify default values for action parts. [待校准@11734]
