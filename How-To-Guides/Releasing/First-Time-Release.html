First Time Release [待校准@792]

This guide explains how to release ROS 2 packages that you have not released before. Due to numerous options available when releasing ROS packages, this guide intends to cover the most common scenario and does not cover every corner-case. [待校准@793]

Be part of a release team [待校准@794]

You must be part of a release team. If you are not part of a release team yet, follow either: [待校准@795]

  • Join a release team [待校准@796]

  • Start a new release team [待校准@797]

Create a new release repository [待校准@798]

You need a release repository to release a package. Follow Create a new release repository. [待校准@799]

Install dependencies [待校准@199]

Install tools that you will use in the upcoming steps according to your platform: [待校准@800]

sudo apt install python3-bloom python3-catkin-pkg

sudo dnf install python3-bloom python3-catkin_pkg

pip3 install -U bloom catkin_pkg

Set Up a Personal Access Token [待校准@804]

警告

If the file ~/.config/bloom exists on your computer, it is likely that you have done this before so you should skip this section. [待校准@805]

During the release process, multiple HTTPS Git operations will be performed that require password authentication. To avoid being repeatedly asked for a password, a Personal Access Token (PAT) will be set up. If you have multi-factor authentication setup on your GitHub account, you must setup a Personal Access Token. [待校准@806]

Create a Personal Access Token by: [待校准@807]

  1. Log in to GitHub and go to Personal access tokens. [待校准@808]

  2. Click the Generate new token button. [待校准@809]

  3. Set Note to something like Bloom token. [待校准@810]

  4. Set Expiration to No expiration. [待校准@811]

  5. Tick the public_repo and workflow checkboxes. [待校准@812]

  6. Click the Generate token button. [待校准@813]

After you have created the token, you will end up back at the Personal access tokens page. Copy the alphanumeric token that is highlighted in green. [待校准@814]

Save your GitHub username and PAT to a new file called ~/.config/bloom, with the format below: [待校准@815]

{
   "github_user": "<your-github-username>",
   "oauth_token": "<token-you-created-for-bloom>"
}

Ensure repositories are up-to-date [待校准@816]

Make sure that: [待校准@817]

  • Your repository is hosted on a remote such as GitHub. [待校准@818]

  • You have a clone of the repository on your computer and are on the right branch. [待校准@819]

  • Both the remote repository and your clone are up-to-date. [待校准@820]

Generate Changelog [待校准@821]

Generate a CHANGELOG.rst file per package in your repo using the following command: [待校准@822]

catkin_generate_changelog --all

Open all CHANGELOG.rst files in an editor. You will see that catkin_generate_changelog has auto-generated a forthcoming section with notes from commit messages: [待校准@823]

^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
Changelog for package your_package
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

Forthcoming
-----------
* you can modify this commit message
* and this

Clean up the list of commit messages to concisely convey the notable changes that have been made to the packages since the last release, and commit all the CHANGELOG.rst files. Do not modify the Forthcoming header. [待校准@824]

Bump the package version [待校准@825]

Every release of the package must have a unique version number higher than the previous release. Run: [待校准@826]

catkin_prepare_release

which performs the following: [待校准@827]

  1. increases the package version in package.xml [待校准@828]

  2. replaces the heading Forthcoming with version (date) (eg. 0.0.1 (2022-01-08)) in CHANGELOG.rst [待校准@829]

  3. commits those changes [待校准@830]

  4. creates a tag (eg. 0.0.1) [待校准@831]

  5. pushes the changes and the tag to your remote repository [待校准@832]

注解

By default the patch version of the package is incremented, such as from 0.0.0 to 0.0.1. To increment the minor or major version instead, run catkin_prepare_release --bump minor or catkin_prepare_release --bump major. For more details, see catkin_prepare_release --help. [待校准@833]

Bloom Release [待校准@834]

Run the following command, replacing my_repo with the name of your repository: [待校准@835]

bloom-release --new-track --rosdistro humble --track humble my_repo

小技巧

  • --new-track tells bloom to create a new track and configure it. [待校准@836]

  • --rosdistro humble indicates that this release is for the humble distro [待校准@837]

  • --track humble indicates that you want the track name to be humble [待校准@838]

You will be prompted to enter information to configure a new track. In a common scenario such as: [待校准@839]

  • Your packages are in a repository called my_repo [待校准@840]

  • You are releasing a branch called main [待校准@841]

  • The repository is hosted on GitHub at https://github.com/my_organization/my_repo.git [待校准@842]

  • Your release repository is at https://github.com/ros2-gbp/my_repo-release.git [待校准@843]

You should respond to the prompts as following: [待校准@844]

Configuration [待校准@845]

Value [待校准@846]

Release Repository url [待校准@847]

https://github.com/ros2-gbp/my_repo-release.git [待校准@848]

Repository Name [待校准@849]

my_repo [待校准@850]

Upstream Repository URI [待校准@851]

https://github.com/my_organization/my_repo.git [待校准@852]

Upstream VCS Type [待校准@853]

Version [待校准@854]

Release Tag [待校准@855]

Upstream Devel Branch [待校准@856]

main [待校准@857]

ROS Distro [待校准@858]

Patches Directory [待校准@859]

Release Repository Push URL [待校准@860]

注解

An empty cell in the table indicates that the default value should be used. Simply respond to the prompt by pressing Enter. [待校准@861]

Bloom will automatically create a pull request for you against rosdistro. [待校准@862]

Next Steps [待校准@863]

Once your pull request has been submitted, usually within one or two days, one of the maintainers of rosdistro will review and merge your Pull Request. If your package build is successful, in 24-48 hours your packages will become available in the ros-testing repository, where you can test your pre-release binaries. [待校准@864]

Approximately every two to four weeks, the distribution's release manager manually synchronizes the contents of ros-testing into the main ROS repository. This is when your packages actually become available to the rest of the ROS community. To get updates on when the next synchronization (sync) is coming, subscribe to the Packaging and Release Management Category on ROS Discourse. [待校准@865]
