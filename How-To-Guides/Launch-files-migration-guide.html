Migrating launch files from ROS 1 to ROS 2 [待校准@10180]

Table of Contents [待校准@9696]

  • Background [待校准@10182]

  • Migrating tags from ROS 1 to ROS 2 [待校准@10184]

  • New tags in ROS 2 [待校准@10252]

  • Replacing an include tag [待校准@10261]

  • Substitutions [待校准@10265]

  • Type inference rules [待校准@10200]

This guide describes how to write XML launch files for an easy migration from ROS 1. [待校准@10181]

Background [待校准@10182]

A description of the ROS 2 launch system and its Python API can be found in Launch System tutorial. [待校准@10183]

Migrating tags from ROS 1 to ROS 2 [待校准@10184]

launch [待校准@10185]

  • Available in ROS 1. [待校准@10186]

  • launch is the root element of any ROS 2 launch XML file. [待校准@10187]

node [待校准@10188]

  • Available in ROS 1. [待校准@10189]

  • Launches a new node. [待校准@10190]

  • Differences from ROS 1: [待校准@10191]

    • type attribute is now exec. [待校准@10192]

    • ns attribute is now namespace. [待校准@10193]

    • The following attributes aren't available: machine, respawn_delay, clear_params. [待校准@10194]

Example [待校准@10195]

<launch>
   <node pkg="demo_nodes_cpp" exec="talker"/>
   <node pkg="demo_nodes_cpp" exec="listener"/>
</launch>

param [待校准@10196]

  • Available in ROS 1. [待校准@10197]

  • Used for passing a parameter to a node. [待校准@10198]

  • There's no global parameter concept in ROS 2. For that reason, it can only be used nested in a node tag. Some attributes aren't supported in ROS 2: type, textfile, binfile, executable, command. [待校准@10199]

Example [待校准@10195]

<launch>
   <node pkg="demo_nodes_cpp" exec="parameter_event">
      <param name="foo" value="5"/>
   </node>
</launch>

Type inference rules [待校准@10200]

Here are some examples of how to write parameters: [待校准@10201]

<node pkg="my_package" exec="my_executable" name="my_node">
   <!--A string parameter with value "1"-->
   <param name="a_string" value="'1'"/>
   <!--A integer parameter with value 1-->
   <param name="an_int" value="1"/>
   <!--A float parameter with value 1.0-->
   <param name="a_float" value="1.0"/>
   <!--A string parameter with value "asd"-->
   <param name="another_string" value="asd"/>
   <!--Another string parameter, with value "asd"-->
   <param name="string_with_same_value_as_above" value="'asd'"/>
   <!--Another string parameter, with value "'asd'"-->
   <param name="quoted_string" value="\'asd\'"/>
   <!--A list of strings, with value ["asd", "bsd", "csd"]-->
   <param name="list_of_strings" value="asd, bsd, csd" value-sep=", "/>
   <!--A list of ints, with value [1, 2, 3]-->
   <param name="list_of_ints" value="1,2,3" value-sep=","/>
   <!--Another list of strings, with value ["1", "2", "3"]-->
   <param name="another_list_of_strings" value="'1';'2';'3'" value-sep=";"/>
   <!--A list of strings using an strange separator, with value ["1", "2", "3"]-->
   <param name="strange_separator" value="'1'//'2'//'3'" value-sep="//"/>
</node>

Parameter grouping [待校准@10202]

In ROS 2, param tags are allowed to be nested. For example: [待校准@10203]

<node pkg="my_package" exec="my_executable" name="my_node" ns="/an_absoulute_ns">
   <param name="group1">
      <param name="group2">
         <param name="my_param" value="1"/>
      </param>
      <param name="another_param" value="2"/>
   </param>
</node>

That will create two parameters: [待校准@10204]

  • A group1.group2.my_param of value 1, hosted by node /an_absolute_ns/my_node. [待校准@10205]

  • A group1.another_param of value 2 hosted by node /an_absolute_ns/my_node. [待校准@10206]

It's also possible to use full parameter names: [待校准@10207]

<node pkg="my_package" exec="my_executable" name="my_node" ns="/an_absoulute_ns">
   <param name="group1.group2.my_param" value="1"/>
   <param name="group1.another_param" value="2"/>
</node>

rosparam [待校准@10208]

  • Available in ROS 1. [待校准@10209]

  • Loads parameters from a yaml file. [待校准@10210]

  • It has been replaced with a from attribute in param tags. [待校准@10211]

Example [待校准@10195]

<node pkg="my_package" exec="my_executable" name="my_node" ns="/an_absoulute_ns">
   <param from="/path/to/file"/>
</node>

remap [待校准@10212]

  • Available in ROS 1. [待校准@10213]

  • Used to pass remapping rules to a node. [待校准@10214]

  • It can only be used within node tags. [待校准@10215]

Example [待校准@10195]

<launch>
   <node pkg="demo_nodes_cpp" exec="talker">
      <remap from="chatter" to="my_topic"/>
   </node>
   <node pkg="demo_nodes_cpp" exec="listener">
      <remap from="chatter" to="my_topic"/>
   </node>
</launch>

include [待校准@10216]

  • Available in ROS 1. [待校准@10217]

  • Allows including another launch file. [待校准@10218]

  • Differences from ROS 1: [待校准@10191]

    • Available in ROS 1, included content was scoped. In ROS 2, it's not. Nest includes in group tags to scope them. [待校准@10219]

    • ns attribute is not supported. See example of push_ros_namespace tag for a workaround. [待校准@10220]

    • arg tags nested in an include tag don't support conditionals (if or unless). [待校准@10221]

    • There is no support for nested env tags. set_env and unset_env can be used instead. [待校准@10222]

    • Both clear_params and pass_all_args attributes aren't supported. [待校准@10223]

Examples [待校准@10224]

See Replacing an include tag. [待校准@10225]

arg [待校准@10226]

  • Available in ROS 1. [待校准@10227]

  • arg is used for declaring a launch argument, or to pass an argument when using include tags. [待校准@10228]

  • Differences from ROS 1: [待校准@10191]

    • value attribute is not allowed. Use let tag for this. [待校准@10229]

    • doc is now description. [待校准@10230]

    • When nested within an include tag, if and unless attributes aren't allowed. [待校准@10231]

Example [待校准@10195]

<launch>
   <arg name="topic_name" default="chatter"/>
   <node pkg="demo_nodes_cpp" exec="talker">
      <remap from="chatter" to="$(var topic_name)"/>
   </node>
   <node pkg="demo_nodes_cpp" exec="listener">
      <remap from="chatter" to="$(var topic_name)"/>
   </node>
</launch>

Passing an argument to the launch file [待校准@10232]

In the XML launch file above, the topic_name defaults to the name chatter, but can be configured on the command-line. Assuming the above launch configuration is in a file named mylaunch.xml, a different topic name can be used by launching it with the following: [待校准@10233]

ros2 launch mylaunch.xml topic_name:=custom_topic_name

There is some additional information about passing command-line arguments in Using Substitutions. [待校准@10234]

env [待校准@10235]

  • Available in ROS 1. [待校准@10236]

  • Sets an environment variable. [待校准@10237]

  • It has been replaced with env, set_env and unset_env: [待校准@10238]

    • env can only be used nested in a node or executable tag. if and unless tags aren't supported. [待校准@10239]

    • set_env can be nested within the root tag launch or in group tags. It accepts the same attributes as env, and also if and unless tags. [待校准@10240]

    • unset_env unsets an environment variable. It accepts a name attribute and conditionals. [待校准@10241]

Example [待校准@10195]

<launch>
   <set_env name="MY_ENV_VAR" value="MY_VALUE" if="CONDITION_A"/>
   <set_env name="ANOTHER_ENV_VAR" value="ANOTHER_VALUE" unless="CONDITION_B"/>
   <set_env name="SOME_ENV_VAR" value="SOME_VALUE"/>
   <node pkg="MY_PACKAGE" exec="MY_EXECUTABLE" name="MY_NODE">
      <env name="NODE_ENV_VAR" value="SOME_VALUE"/>
   </node>
   <unset_env name="MY_ENV_VAR" if="CONDITION_A"/>
   <node pkg="ANOTHER_PACKAGE" exec="ANOTHER_EXECUTABLE" name="ANOTHER_NODE"/>
   <unset_env name="ANOTHER_ENV_VAR" unless="CONDITION_B"/>
   <unset_env name="SOME_ENV_VAR"/>
</launch>

group [待校准@10242]

  • Available in ROS 1. [待校准@10243]

  • Allows limiting the scope of launch configurations. Usually used together with let, include and push_ros_namespace tags. [待校准@10244]

  • Differences from ROS 1: [待校准@10191]

    • There is no ns attribute. See the new push_ros_namespace tag as a workaround. [待校准@10245]

    • clear_params attribute isn't available. [待校准@10246]

    • It doesn't accept remap nor param tags as children. [待校准@10247]

Example [待校准@10195]

launch-prefix configuration affects both executable and node tags' actions. This example will use time as a prefix if use_time_prefix_in_talker argument is 1, only for the talker. [待校准@10248]

<launch>
   <arg name="use_time_prefix_in_talker" default="0"/>
   <group>
      <let name="launch-prefix" value="time" if="$(var use_time_prefix_in_talker)"/>
      <node pkg="demo_nodes_cpp" exec="talker"/>
   </group>
   <node pkg="demo_nodes_cpp" exec="listener"/>
</launch>

New tags in ROS 2 [待校准@10252]

set_env and unset_env [待校准@10253]

See env tag decription. [待校准@10254]

push_ros_namespace [待校准@10255]

include and group tags don't accept an ns attribute. This action can be used as a workaround: [待校准@10256]

<!-Other tags-->
<group>
   <push_ros_namespace namespace="my_ns"/>
   <!--Nodes here are namespaced with "my_ns".-->
   <!--If there is an include action here, its nodes will also be namespaced.-->
   <push_ros_namespace namespace="another_ns"/>
   <!--Nodes here are namespaced with "another_ns/my_ns".-->
   <push_ros_namespace namespace="/absolute_ns"/>
   <!--Nodes here are namespaced with "/absolute_ns".-->
   <!--The following node receives an absolute namespace, so it will ignore the others previously pushed.-->
   <!--The full path of the node will be /asd/my_node.-->
   <node pkg="my_pkg" exec="my_executable" name="my_node" ns="/asd"/>
</group>
<!--Nodes outside the group action won't be namespaced.-->
<!-Other tags-->

It's a replacement of arg tag with a value attribute. [待校准@10258]

<let name="foo" value="asd"/>

executable [待校准@10259]

It allows running any executable. [待校准@10260]

<executable cmd="ls -las" cwd="/var/log" name="my_exec" launch-prefix="something" output="screen" shell="true">

</executable>

Replacing an include tag [待校准@10261]

In order to include a launch file under a namespace as in ROS 1 then the include tags must be nested in a group tag. [待校准@10262]

<group>
   <include file="another_launch_file"/>
</group>

Then, instead of using the ns attribute, add the push_ros_namespace action tag to specify the namespace: [待校准@10263]

<group>
   <push_ros_namespace namespace="my_ns"/>
   <include file="another_launch_file"/>
</group>

Nesting include tags under a group tag is only required when specifying a namespace [待校准@10264]

Substitutions [待校准@10265]

Documentation about ROS 1's substitutions can be found in roslaunch XML wiki. Substitutions syntax hasn't changed, i.e. it still follows the $(substitution-name arg1 arg2 ...) pattern. There are, however, some changes w.r.t. ROS 1: [待校准@10266]

  • env and optenv tags have been replaced by the env tag. $(env <NAME>) will fail if the environment variable doesn't exist. $(env <NAME> '') does the same as ROS 1's $(optenv <NAME>). $(env <NAME> <DEFAULT>) does the same as ROS 1's $(env <NAME> <DEFAULT>) or $(optenv <NAME> <DEFAULT>). [待校准@10267]

  • find has been replaced with find-pkg-share (substituting the share directory of an installed package). Alternatively find-pkg-prefix will return the root of an installed package. [待校准@10268]

  • There is a new exec-in-pkg substitution. e.g.: $(exec-in-pkg <package_name> <exec_name>). [待校准@10269]

  • There is a new find-exec substitution. [待校准@10270]

  • arg has been replaced with var. It looks at configurations defined either with arg or let tag. [待校准@10271]

  • eval and dirname substitutions haven't changed. [待校准@10272]

  • anon substitution is not supported. [待校准@10273]

Type inference rules [待校准@10200]

The rules that were shown in Type inference rules subsection of param tag applies to any attribute. For example: [待校准@10274]

<!--Setting a string value to an attribute expecting an int will raise an error.-->
<tag1 attr-expecting-an-int="'1'"/>
<!--Correct version.-->
<tag1 attr-expecting-an-int="1"/>
<!--Setting an integer in an attribute expecting a string will raise an error.-->
<tag2 attr-expecting-a-str="1"/>
<!--Correct version.-->
<tag2 attr-expecting-a-str="'1'"/>
<!--Setting a list of strings in an attribute expecting a string will raise an error.-->
<tag3 attr-expecting-a-str="asd, bsd" str-attr-sep=", "/>
<!--Correct version.-->
<tag3 attr-expecting-a-str="don't use a separator"/>

Some attributes accept more than a single type, for example value attribute of param tag. It's usual that parameters that are of type int (or float) also accept an str, that will be later substituted and tried to convert to an int (or float) by the action. [待校准@10275]
